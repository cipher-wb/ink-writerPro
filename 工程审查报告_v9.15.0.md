## 一、项目目标与成功标准

**项目定位**：长篇网文工业化创作系统，作为 Claude Code 插件运行，通过 14 个 AI Agent + 13 个 Skill + 数据链路 + RAG 将 200 万字规模小说的创作自动化。

**成功标准**（从工程视角推导）：

| 维度 | 标准 |
|------|------|
| 可靠性 | 状态数据（state.json + index.db）不丢失、不错乱 |
| 可维护性 | 模块边界清晰，新 Agent/Skill 可独立开发和测试 |
| 可复现性 | CI 绿 = 本地绿 = 发布安全 |
| 跨版本兼容 | v8 项目可在 v9 下运行，迁移可选 |

**判定：当前架构基本支撑上述目标。** 模块拆分合理（scripts / dashboard / agents / skills / genres 五层分离），Mixin 组合模式管理了 IndexManager 的复杂度，双存储 + 原子写入保障了数据安全。以下报告聚焦剩余的结构性问题。

## 二、发现的问题

### 问题 1：Lock 文件来源声明与实际内容不一致

| 字段 | 值 |
|------|-----|
| **严重级别** | 🔴 高 |
| **证据** | `ink-writer/scripts/requirements.lock` 头部声明 `pip-compile ... ink-writer/scripts/requirements.txt`（仅含 aiohttp/filelock/pydantic），但实际包含 `pytest==9.0.2`、`pytest-asyncio==1.3.0`、`pytest-cov==7.1.0`、`pytest-timeout==2.4.0` 共 4 个测试依赖 |
| **影响** | 1) 发布流水线 `plugin-release.yml` 用此 lock 文件安装依赖后跑测试——**表面可行但逻辑错误**：lock 文件语义是"生产依赖的精确快照"，混入测试依赖后，无法区分生产环境真正需要的包。2) 如果未来有人基于此 lock 做最小化部署，会多装 4 个不必要的包。3) 头部的 compile 命令已不可重现（用它重新编译不会产出含 pytest 的结果） |
| **建议** | 拆分为两个 lock 文件：`requirements.lock`（编译自 requirements.txt，纯生产）和 `requirements-dev.lock`（编译自 requirements-dev.txt，含测试）。CI 测试安装 dev.lock，发布流水线安装 prod lock + dev.lock |

---

### 问题 2：v9.14.0 和 v9.15.0 已提交但未打 tag/release

| 字段 | 值 |
|------|-----|
| **严重级别** | 🟡 中 |
| **证据** | 代码中版本号为 `9.15.0`（plugin.json / marketplace.json / \_\_init\_\_.py / package.json 均一致），但 `git tag --sort=-v:refname` 最新为 `v9.13.0`。README 变更日志列有 v9.14.0 和 v9.15.0 条目 |
| **影响** | 1) 插件缓存目录 `8.2.0` 指向开发目录（符号链接），用户实际运行的版本与最后发布版本有 2 个版本差。2) 如果此时有其他用户通过 marketplace 安装，获取的是 v9.13.0 而非最新代码 |
| **建议** | 执行 `plugin-release.yml` workflow dispatch 分别发布 v9.14.0 和 v9.15.0，或合并为一次 v9.15.0 发布 |

---

### 问题 3：20 个测试文件手动操作 sys.path

| 字段 | 值 |
|------|-----|
| **严重级别** | 🟡 中 |
| **证据** | 46 个测试文件中有 20 个包含 `sys.path.insert(0, ...)` 来定位 `scripts/` 或 `dashboard/` 目录，尽管 `pytest.ini` 已声明 `pythonpath = ink-writer/scripts` |
| **影响** | 1) 每个测试文件维护自己的路径解析逻辑，存在路径计算错误风险（如 `parents[2]` vs `parents[3]` 取决于文件深度）。2) 与 pytest.ini 的 `pythonpath` 配置语义重复，增加理解成本。3) 如果目录结构重构，需逐个修改 |
| **建议** | 在 `pytest.ini` 的 `pythonpath` 中追加 `ink-writer/dashboard`，然后移除所有测试文件中的 `sys.path.insert` 手动操作。对于 `scripts/` 上层模块（workflow_manager、migration_auditor 等），统一通过 conftest.py 或 pytest pythonpath 解决 |

---

### 问题 4：发布流水线仅在 Ubuntu 运行，不覆盖 macOS

| 字段 | 值 |
|------|-----|
| **严重级别** | 🟡 中 |
| **证据** | `ci-test.yml` 矩阵含 `[ubuntu-latest, macos-latest]`，但 `plugin-release.yml` 硬编码 `runs-on: ubuntu-latest`，发布前的测试仅在 Linux 上执行 |
| **影响** | macOS 是主要使用平台（用户明确声明只关注 mac 版本），但发布门禁不包含 macOS 测试。如果存在 macOS 特有的路径处理或 SQLite 行为差异，可能在发布后才暴露 |
| **建议** | 方案一（推荐）：在 `plugin-release.yml` 的 release job 前增加一个 `test` job，复用 ci-test.yml 的矩阵（或至少包含 macOS + Python 3.14），release job 设置 `needs: [test]`。方案二：发布前要求 CI Test workflow 在当前 commit 已通过（通过 GitHub required status checks 保障） |

---

### 问题 5：缺少静态分析和格式化工具链

| 字段 | 值 |
|------|-----|
| **严重级别** | 🟢 低 |
| **证据** | 仓库中无 `ruff.toml`、`mypy.ini`、`.flake8`、`pyproject.toml`（用于 black/isort 配置）、`.pre-commit-config.yaml` 等文件。质量门禁仅依赖 pytest + 70% 覆盖率阈值 |
| **影响** | 代码风格完全依赖开发者自觉。随着模块增长（当前 scripts 目录 82 个 .py 文件），风格漂移、未使用 import、类型错误等只能靠人工发现 |
| **建议** | 引入 ruff（集成 lint + format，零配置即可启用），加入 CI 流水线。暂不建议引入 mypy（项目大量使用动态特性和 Claude Code 插件协议，类型标注成本较高） |

---

### 问题 6：Dashboard 测试依赖路径不独立

| 字段 | 值 |
|------|-----|
| **严重级别** | 🟢 低 |
| **证据** | Dashboard 模块有自己的 `requirements.txt` 和 `requirements.lock`，但没有 `requirements-dev.txt`。Dashboard 的测试文件（test_dashboard_app.py 等）位于 `scripts/data_modules/tests/` 目录下，而非 dashboard 自身目录中。测试通过 `sys.path.insert` 手动引入 dashboard 路径 |
| **影响** | Dashboard 无法作为独立子模块运行测试；其测试依赖于 scripts 的 pytest 配置和路径。如果未来 Dashboard 需要独立开发或部署，需要额外拆分工作 |
| **建议** | 当前阶段保持现状即可（项目规模和团队规模不大）。如果 Dashboard 未来独立发展，再考虑拆出独立的 tests/ 和 dev requirements |

## 三、架构亮点（无需改动）

| 方面 | 说明 |
|------|------|
| **Mixin 组合** | IndexManager 通过 5 个 Mixin 拆分职责（Chapter/Entity/Debt/Reading/Observability），2068 行代码被合理分解 |
| **懒加载导出** | `data_modules/__init__.py` 使用 `__getattr__` + `_LAZY_EXPORTS` 延迟导入 30+ 类，插件加载快 |
| **双存储同步** | state.json（人类可读）+ index.db（可查询）通过 SQLStateManager 原子同步，设计合理 |
| **版本同步脚本** | `sync_plugin_version.py` 统一管理 7 个版本声明点，CI 自动校验一致性 |
| **渐进式检查点** | 每 5/10/20 章触发不同级别审查，成本可控 |
| **依赖分层** | 生产 vs 开发依赖已拆分（requirements.txt vs requirements-dev.txt），只需修正 lock 文件 |
| **优雅降级** | jieba 可选，RAG 可回退到 BM25，API 不可用时系统仍能工作 |
| **测试覆盖率** | 1021 个测试，覆盖率 84.50%，远超 70% 阈值 |

## 四、总体评价

### 合理 ✅

项目架构设计与其目标（长篇网文工业化创作）匹配度高。核心数据链路（state.json + index.db + vectors.db）、Agent 协调机制、渐进检查点、版本管理均有完整方案。发现的问题均为工程卫生类，不涉及架构层面的设计缺陷。

## 五、最关键的 3 个问题

| 排序 | 问题 | 原因 |
|------|------|------|
| 1 | Lock 文件声明与内容不一致 | 破坏了"lock 文件 = 可复现生产环境"的语义契约，且头部命令不可重现 |
| 2 | 发布流水线不覆盖 macOS | 主使用平台是 macOS，但发布门禁只在 Linux 验证 |
| 3 | 两个版本未发布 | 代码已就绪但 tag/release 缺失，用户可能运行过期版本 |

## 六、最值得做的 3 个优化

| 排序 | 优化 | 预期收益 | 工作量 |
|------|------|----------|--------|
| 1 | 拆分 lock 文件（prod vs dev） | 恢复依赖语义正确性，发布安全 | 小（重新编译两次 pip-compile） |
| 2 | 消除测试中的 sys.path.insert | 减少 20 个文件的维护债务，降低重构风险 | 中（需逐个验证 import 正确性） |
| 3 | 引入 ruff lint | 统一代码风格，自动捕获未使用 import 和基础错误 | 小（零配置可启动，逐步收紧规则） |

## 七、先别乱动的地方

| 区域 | 原因 |
|------|------|
| **IndexManager Mixin 结构** | 当前 5 个 Mixin 拆分已经合理，不要尝试进一步拆细或合并 |
| **state.json + index.db 双存储** | 同步机制已稳定运行，贸然改为单存储会破坏大量下游逻辑 |
| **懒加载 `__init__.py`** | 当前 `__getattr__` 方案工作正常，不要换成 eager import 或 `__init_subclass__` |
| **data_modules/tests/ 目录位置** | 虽然 dashboard 测试混在 scripts 测试中不够理想，但现在拆分成本高于收益 |
| **genre/ 模板系统** | 38 个流派模板是内容资产，结构简单稳定，无需工程化改造 |
