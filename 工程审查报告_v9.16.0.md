## 三、发现的新问题

### 问题 1：context_manager.py 中 `except Exception` 密度过高（14 处）

| 字段 | 值 |
|------|-----|
| **严重级别** | 🟡 中 |
| **证据** | `context_manager.py`（1317 行）包含 14 处 `except Exception`，为全项目单文件最多。其中约 5 处（canary 查询，行 339/363/387/413/415）因表可能不存在而 `pass` 是合理设计；但另外 9 处存在问题：行 580（memory card 保存失败静默吞掉）���行 1086/1139（角色演变/关系轨迹查询失败静默 return）、行 1202（大纲文件读取失败静默 continue）、行 1235（故事骨架高分章节查询失败静默降级）等，均不记录日志 |
| **影响** | 当 IndexManager 接口变更或 index.db schema 迁移出 bug 时，context_manager 不会报任何错误，而是静默返回空数据。调试时需要逐个加断点排查，排错成本高 |
| **建议** | 对非 canary 的 `except Exception` 补充 `logger.debug` ��� `logger.warning`，至少记录一行日志。canary 部分可保持现状（已有注释说明设计意图） |

### 问题 2：SQLite 连接泄漏（测试中可复现 ResourceWarning）

| 字段 | 值 |
|------|-----|
| **严重级别** | 🟡 中 |
| **证据** | 执行 `python3 -m pytest` 产生 9 条 `ResourceWarning: unclosed database in <sqlite3.Connection>`，来源为 `ast.py:46`（Python 3.14 的模块导入路径）和 `importlib._bootstrap_external:511`。追踪到 `index_manager.py` 中 `IndexManager.__init__` 初始化时会调用 `self._ensure_tables()` 打开连接，但若初始化中途被 GC 或测试 fixture 销毁，连接未被显式 close |
| **影响** | 1) 在长时间运行的批量写作场景（ink-auto 5+章）中，未关闭的 SQLite 连接可能导致 macOS 上的文件句柄累积。2) Python 3.14 对 ResourceWarning 更严格，CI 日志中出现大量警告降低信噪比。3) 虽然 `context_manager.py:93` 和 `rag_adapter.py:254` 已使用 `@contextmanager` 包装连接，但 `index_manager.py` 主类未采用相同模式 |
| **建议** | 为 `IndexManager` 添加 `close()` 方法和 `__del__` 安全网（或实现 context manager 协议），在需要短连接的场景使用 `with` 语句。测试 fixture 中显式调用 `index_mgr.close()` |

### 问题 4：`state_manager._sync_to_sqlite()` 中重复创建 IndexManager 实例

| 字段 | 值 |
|------|-----|
| **严重级别** | 🟡 中 |
| **证据** | `state_manager.py:490` 在 `_sync_to_sqlite()` 方法内 `from .index_manager import IndexManager; idx = IndexManager(self.config)` 每次调用都创建新实例。`StateManager` 本身已通过 `SQLStateManager` 持有一个 `IndexManager`（`sql_state_manager.py:23` 导入），但这里另建了一个 |
| **影响** | 1) 每次 `flush()` → `_sync_to_sqlite()` 都会重新打开 index.db 并执行 schema 检查（`_ensure_tables`），在批量写作时产生不必要的 I/O 开销。2) 两个 IndexManager 实例可能持有不同的连接/缓存状态，理论上不会出错（SQLite WAL 模式支持并发读），但增加了理解复杂度 |
| **建议** | 复用 `self._sql_state_manager._index_manager`（如果已暴露）或在 `StateManager.__init__` 中创建单一 `IndexManager` 实例并在两处共享 |

### 问题 5：全项目仍有 84 处 `except Exception`

| 字段 | 值 |
|------|-----|
| **严重级别** | 🟢 低 |
| **证据** | `grep -rn "except Exception" scripts/ --include="*.py"` 统计 84 处，分布在 22 个文件中。v9.14.0 已精确化了部分（如 `state_manager.py` 现使用 `except (OSError, sqlite3.OperationalError)` + `except Exception` 分级模式），但总量仍高。高频文件：context_manager(14)、rag_adapter(8)、project_locator(7)、migrate_state_to_sqlite(8)、computational_checks(6)、extract_chapter_context(6) |
| **影响** | 大部分属于 CLI 入口/日志降级/表可能不存在 三类合理场景，真正有问题的约 15~20 处（静默 pass 且不记录日志）。问题不在数量而在缺少日志记录 |
| **建议** | 不建议逐个精确化（工作量大且部分场景确实需要兜底），建议对所有 `except Exception: pass` 模式补充 `logger.debug`，确保可追踪 |

---

### 问题 6：`computational_checks.py` 中 SQLite 连接未使用 context manager

| 字段 | 值 |
|------|-----|
| **严重级别** | 🟢 低 |
| **证据** | `computational_checks.py:159` 使用 `conn = sqlite3.connect(str(db_path))` 后手动 `conn.close()`（行 177），但中间有 `try/except` 结构。若在 `cursor.execute()` 和 `conn.close()` 之间抛出非预期异常（如 KeyboardInterrupt），连接不会关闭 |
| **影响** | 理论风险低（computational_checks 是短命脚本，进程退出后 OS 释放句柄），但与 `context_manager.py` 和 `rag_adapter.py` 的 `@contextmanager` + `with` 模式不一致 |
| **建议** | 改为 `with closing(sqlite3.connect(...)) as conn:` 模式，统一风格 |
