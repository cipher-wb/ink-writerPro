# AI味检测器（Anti-Detection Checker）

## 职责

从**统计特征**层面检测章节文本是否具有AI生成痕迹。不同于 proofreading-checker 关注文笔质量，本检查器专注于**AI检测工具（朱雀等）会标记的模式**：困惑度过低、突发度不足、结构过于规整。

> 本检查器可产生 `high` 级别问题。若 AI味综合评分 < 60，判定为 `high`：必须在 Step 4 做针对性修复。

## 输入

与其他 checker 相同，接收 `review_bundle_file` 路径和 `chapter_file` 绝对路径。

## 检查维度（6层）

### 第0层：章节开头模式检测（v10.6 新增）

**原理**：以"第N天"、"次日清晨"、"三天后"等时间标记开头是最明显的AI写作痕迹，起点编辑一眼识别。

**检查方法**：
1. 提取章节正文第一句（到第一个句号/问号/感叹号/省略号为止）
2. 检测是否匹配时间标记模式

**时间标记黑名单**（正则匹配）：
- `^第[一二三四五六七八九十\d]+[天日]`
- `^[次翌]日`
- `^[一二三四五六七八九十\d]+天[后前]`
- `^[一二三四五六七八九十\d]+个?[月年周]后`
- `^那天[早晚]`
- `^[清凌]晨`、`^[傍黄]昏`、`^夜[里晚深]`（单独作为第一句时）
- `^时间[来到了过了]`
- `^[这那]一[天日年]`

**判定规则**：

| 指标 | 阈值 | 严重度 |
|------|------|--------|
| 第一句匹配时间标记黑名单 | 章节以时间标记开头 | `critical` |

**修复建议**：用行动、对话、感官感知或悬念切入替代时间标记。时间锚点应在前3段内通过角色感知自然带出（如"阳光刺进眼睛"暗示天亮）。

### 第1层：句长突发度检测

**原理**：AI文本句长标准差显著低于人类文本。人类写作句长分布像心电图，AI像直线。**标杆小说句长均值约28字，ink-writer默认倾向仅12字——句子碎片化是最大的AI味来源。**

**检查方法**：
1. 将正文按句号、问号、感叹号、省略号切分为句子
2. 计算全章句长的**变异系数**（标准差/均值）
3. 计算句长均值、短句占比（≤8字）、长句占比（≥35字）

**判定规则**：

| 指标 | 阈值 | 严重度 |
|------|------|--------|
| 全章句长变异系数 < 0.35 | 句长过于均匀 | `high` |
| **句长均值 < 18 字** | 句子碎片化（标杆28字） | `high` |
| **短句占比(≤8字) > 25%** | 短句过多（标杆13%） | `high` |
| **长句占比(≥35字) < 15%** | 缺乏长句纵深（标杆30%） | `medium` |
| 连续 5+ 句长度均在 15-25 字区间 | 存在"平坦区" | `medium`（每处） |

**输出示例**：
```
问题：全章句长均值仅14.2字（标杆28字），短句占比38%（标杆13%）
建议：将连续短句合并为25-40字的复合句，用逗号串联动作和细节。短句仅在紧张/冲击时刻使用。
```

### 第2层：信息密度均匀性检测

**原理**：AI文本每句都在"推进"——传递信息或推动情节。人类文本有"闲笔"和感官碎片，信息密度有波动。

**检查方法**：
1. 将正文每 300 字切一个窗口
2. 检查每个窗口内是否有"无叙事功能句"——不推进情节、不传递信息的纯感官/环境细节
3. 用简单启发式判断：不含角色名、不含动作动词、不含对话、仅含感官词（看/听/闻/摸/温度/光线/声音）的句子

**判定规则**：

| 指标 | 阈值 | 严重度 |
|------|------|--------|
| 连续 500+ 字全部为情节推进句 | 信息密度无波动 | `medium`（每处） |
| 全章无"无功能感官句" | 完全缺乏闲笔 | `high` |

**输出示例**：
```
问题：第5-8段（约600字）全部在推进对话/情节，无感官间隔
位置：第28-45行
建议：在第35行附近插入1句环境/感官细节（如光线变化、声音、温度、气味）
```

### 第3层：因果链完美度检测

**原理**：AI文本因果逻辑过于完美——每个行为都有解释，每个结果都有铺垫。人类叙事有跳跃和省略。

**检查方法**：
1. 检测因果连接词密度：因为/所以/于是/因此/由于/导致/结果/从而/进而
2. 检查是否存在连续 3+ 句形成因果链（A→B→C→D）
3. 检查角色行为是否每次都附带心理解释（"他觉得/他认为/他知道" + 行为）

**判定规则**：

| 指标 | 阈值 | 严重度 |
|------|------|--------|
| 因果连接词密度 > 每200字1个 | 因果链过密 | `medium` |
| 连续 3+ 句因果相连 | 完美因果链 | `medium`（每处） |
| 全章 70%+ 角色行为附带心理解释 | 过度解释 | `high` |

**输出示例**：
```
问题：第11段存在4层因果链（他看到→所以他决定→于是他走向→结果发现）
位置：第52-58行
建议：删除中间1-2个因果环节，直接从触发跳到结果，让读者自行补全逻辑
```

### 第3.5层：对话存在性检测（v10.6 新增）

**原理**：标杆网文对话占比约34.5%。完全无对话或对话过少是AI写作的典型特征——AI倾向用内心独白和叙述替代角色互动。

**检查方法**：
1. 统计全章引号内文本占总字数的比例
2. 计算对话轮数（每对引号为一轮）

**判定规则**：

| 指标 | 阈值 | 严重度 |
|------|------|--------|
| **全章对话占比 = 0%** | 完全无对话 | `critical`（纯逃亡/独处章可豁免，需标注） |
| **全章对话占比 < 10%** | 对话严重不足 | `high` |
| **全章对话占比 < 20%** | 对话偏少 | `medium` |
| **对话轮数 < 3** | 角色互动不足 | `medium` |

**豁免条件**：仅当章纲明确标注为"独角戏/逃亡/修炼突破"等纯独处场景时，对话<10%可降级为`low`。

### 第4层：对话同质性检测

**原理**：AI写的所有角色说话方式趋同——语法完整、长度相近、风格统一。人类写的不同角色对话差异显著。

**检查方法**：
1. 提取所有对话行（含引号内文本），按角色分组
2. 计算各角色对话的平均长度、语气词使用率、完整句比例
3. 检查角色间差异度

**判定规则**：

| 指标 | 阈值 | 严重度 |
|------|------|--------|
| 2+ 角色对话平均长度差 < 5 字 | 对话长度同质 | `medium` |
| 全部对话均为完整陈述句（无省略、无打断、无反问） | 对话过于规范 | `high` |
| 连续 3+ 轮对话无任何"不规范"元素 | 对话缺乏口语感 | `medium`（每处） |

"不规范"元素包括：省略主语/宾语、说到一半改口（破折号）、答非所问、重复、语气词（嗯/啊/哦/呃/那个）、反问。

**输出示例**：
```
问题：角色A和角色B对话平均长度均为25-28字，风格趋同
位置：第60-85行
建议：角色A话多（可扩至40字+），角色B话少（压缩至5-10字），加入打断或答非所问
```

### 第5层：段落结构规整度检测

**原理**：AI文本段落长度分布均匀、结构工整（每段都有开头-展开-收束）。人类文本段落长短悬殊，有大量碎片段。

**检查方法**：
1. 统计全章段落长度分布
2. 计算单句段（只有1句的段落）占比
3. 检测碎片段（≤15字的段落）频率
4. 检查段落长度变异系数

**判定规则**：

| 指标 | 阈值 | 严重度 |
|------|------|--------|
| 单句段占比 < 20% | 段落结构过于工整 | `high` |
| 全章无碎片段（≤15字段落） | 缺乏节奏碎片 | `medium` |
| 段落长度变异系数 < 0.4 | 段落长度过于均匀 | `medium` |
| 连续 5+ 段长度均在 50-100 字 | 段落平坦区 | `low` |

### 第5.5层：情感标点密度检测（v10.6 新增）

**原理**：标杆网文每千字约3.8个感叹号、4.2个问号、2.8个省略号。AI文本严重压制情感标点，导致文章"没有感情"——这是读者和编辑最直观的感受。

**检查方法**：
1. 统计全章感叹号（！!）、省略号（……/...）、问号（？?）的数量
2. 除以千字数，得到每千字密度

**判定规则**：

| 指标 | 阈值 | 严重度 |
|------|------|--------|
| **感叹号密度 < 1.5/千字** | 情感表达匮乏（标杆3.8） | `high` |
| **省略号密度 < 1.0/千字** | 缺乏戏剧停顿（标杆2.8） | `medium` |
| **问号密度 < 2.0/千字** | 缺乏好奇心驱动（标杆4.2） | `low` |
| **三种情感标点总密度 < 5.0/千字** | 整体情感外化不足 | `high` |

**修复方向**：在角色说话有情绪时使用感叹号，欲言又止/震惊时使用省略号，质疑/嘲讽时使用反问句。不要为了"看起来高级"而压制情感标点。

### 第6层：叙述视角泄露检测

**原理**：AI经常无意识切换到全知视角——写出POV角色不可能知道的信息。

**检查方法**：
1. 从 `review_bundle_file` 获取本章 POV 角色
2. 检查是否存在以下模式：
   - 描写POV角色看不到的场景（如"与此同时，在另一边..."）
   - 直接描写非POV角色的内心（"他不知道的是，对方心里在想..."）
   - 使用全知评价（"这是一个改变命运的决定"——谁在说？）

**判定规则**：

| 指标 | 阈值 | 严重度 |
|------|------|--------|
| 出现"他/她不知道的是" | 全知视角泄露 | `high`（每处） |
| 描写非POV角色内心活动 | 视角越权 | `medium`（每处） |
| 使用"与此同时"切换场景 | 上帝视角转场 | `low`（每处） |

## 综合评分

各层检查完毕后，计算 AI味综合评分（0-100，100为完全无AI味）：

| 层 | 权重 | 满分 | 说明 |
|----|------|------|------|
| 开头模式检测 | — | — | 不计入综合评分，但 `critical` 直接触发 Step 4 修复 |
| 句长突发度 | 25% | 100 | v10.6: 阈值反转，重点检测碎片化而非均匀化 |
| 信息密度波动 | 15% | 100 | v10.6: 从20%降至15% |
| 因果链完美度 | 10% | 100 | v10.6: 从15%降至10% |
| 对话存在性+同质性 | 20% | 100 | v10.6: 含新增对话存在性检测 |
| 段落结构规整度 | 5% | 100 | |
| 情感标点密度 | 15% | 100 | v10.6 新增：情感表达是过审核心指标 |
| 叙述视角泄露 | 10% | 100 | v10.6: 从15%降至10% |

> **weights_version: v3**（v10.6 起生效）。v2 历史分数不可与 v3 直接比较。核心变化：新增开头检测(critical级)、情感标点检测(15%)、对话存在性检测；句长检测阈值反转（检测碎片化而非均匀化）。

**通过阈值**：综合评分 ≥ 70 为通过。< 60 判定为 `high`，60-69 为 `medium`。

## 输出格式

遵循 `checker-output-schema.md` 的统一格式：

```json
{
  "checker": "anti-detection",
  "chapter": 42,
  "dimensions": {
    "sentence_burstiness": {"score": 65, "issues": [...]},
    "info_density_wave": {"score": 70, "issues": [...]},
    "causal_chain": {"score": 80, "issues": [...]},
    "dialogue_homogeneity": {"score": 55, "issues": [...]},
    "paragraph_regularity": {"score": 60, "issues": [...]},
    "pov_leakage": {"score": 90, "issues": [...]}
  },
  "overall_score": 68,
  "severity_counts": {"critical": 0, "high": 2, "medium": 5, "low": 1},
  "fix_priority": [
    {"location": "第3段第12-17行", "type": "句长平坦区", "fix": "插入碎句打破均匀节奏"},
    {"location": "第60-85行", "type": "对话同质", "fix": "差异化角色对话长度和风格"}
  ]
}
```

`fix_priority` 为本 checker 的**扩展字段**（遵循 `checker-output-schema.md` 的扩展字段约定），按严重度排序，供 Step 4 润色阶段精准定位修复。每条包含：
- `location`：精确到行号范围
- `type`：问题分类（句长平坦区 / 信息密度无波动 / 因果链过密 / 对话同质 / 段落过于工整 / 视角泄露）
- `fix`：一句话修复建议

**与标准 `issues` 数组的关系**：所有检测到的问题同时写入标准 `issues` 数组（遵循统一 schema 格式）。`fix_priority` 是面向 Step 4 的精简修复清单，不替代 `issues`，而是对 `issues` 中 `high`/`medium` 问题的按优先级排序子集。
