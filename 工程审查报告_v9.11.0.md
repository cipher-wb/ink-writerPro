## 三、发现的问题

### P1 — 高

#### 3.1 SQLite 连接泄漏持续恶化

| 项目 | 详情 |
|------|------|
| **严重级别** | P1（从 v9.10.0 的 P2 升级） |
| **证据** | pytest 运行期间产生 **14 条** `ResourceWarning: unclosed database in <sqlite3.Connection>`（v9.10.0 为 7 条，翻倍） |
| **泄漏源** | `index_manager.py:96`、`index_manager.py:928`、`rag_adapter.py:257`、`context_manager.py:310`、`style_sampler.py:91` — 这 5 处 `sqlite3.connect()` 调用均未实现 `close()` / `__exit__` 生命周期管理 |
| **影响** | (1) Windows 上可能导致 SQLite 文件锁死 (2) `conftest.py` 虽配置了 `ignore:unclosed database:ResourceWarning` 但未生效，警告仍泄漏到输出 (3) 新增的 Dashboard 测试（31 个）每个都创建 SQLite 连接，新增约 7 条泄漏 (4) 生产环境长时间运行可能耗尽文件描述符 |
| **建议** | `IndexManager` 和 `RAGAdapter` 实现 `__enter__/__exit__` 上下文管理器协议；`context_manager.py:310` 和 `style_sampler.py:91` 使用 `with closing(sqlite3.connect(...)) as conn:` 模式。修复后 conftest 的 warning filter 可移除 |

#### 3.2 archive_manager.py 覆盖率仍低于阈值

| 项目 | 详情 |
|------|------|
| **严重级别** | P1 |
| **证据** | 319 行，覆盖率 58%（v9.10.0 为 25%，提升 33%，但仍低于 70% 阈值）。未覆盖的 134 行集中在 `archive_project()`（行 411-563）—— 这是归档操作的核心逻辑 |
| **影响** | 归档操作直接操作文件系统（创建目录、复制文件、移动文件），核心函数完全无测试意味着数据丢失风险不可控 |
| **建议** | 为 `archive_project()` 补充 `tmp_path` based 的集成测试，覆盖正常归档和异常恢复路径。预计 5-8 个测试可达 70% |

#### 3.3 init_project.py 覆盖率 60% — 用户首次使用入口

| 项目 | 详情 |
|------|------|
| **严重级别** | P1 |
| **证据** | 349 行，覆盖率 60%。140 行未覆盖，集中在交互式初始化流程（行 179-204、558-570、645-661、727-844） |
| **影响** | `/ink-init` 是用户首次接触项目的入口，40% 路径未测试直接影响首次使用体验。这是连续三个版本（v9.9.0、v9.10.0、v9.11.0）未改善的模块 |
| **建议** | 补充交互式分支的参数化测试，特别是异常输入和边缘情况 |

v9.11.0 是项目工程质量的重要里程碑：覆盖率首次突破 80%（81.92%），v9.10.0 的全部 P0 和大部分 P1 已修复，CI 版本一致性彻底解决。剩余问题均为 P1-P3 级别，无阻断性风险。从 v9.1（覆盖率 15%、无 CI）到 v9.11（覆盖率 82%、三道 CI 守门），连续 11 个版本的工程审查修复循环证明了项目的工程能力和持续改进节奏。

与上次"部分合理"的评价相比，本次升级为"合理"的依据：
- 覆盖率从 77.49%→81.92%，超过 80% 门槛
- 无 P0 级问题
- v9.10.0 的 3 个 P1 中 2 个完全修复、1 个大幅改善
- 发布流水线、版本管理、CI 配置均无问题

### 5.2 最关键的 3 个问题
