| 维度 | 目标 |

## 二、架构评估

### 2.1 整体架构

```
Skills (14) ──→ CLI Router (ink.py) ──→ data_modules/ (领域逻辑)
    ↕                                        ↕
Agents (14)                            SQLite / JSON / Vectors
    ↕
Checkers (10) + Writer + Polish + Context + Data
```

**优点**:
- Skill / Agent / Script 三层分离，职责清晰
- 数据链路（state → index → RAG）有明确管道
- Checkpoint 策略分 5/10/20 章三档，粒度合理
- 版本同步覆盖 4 个平台（Claude Code / Gemini / Codex / Marketplace）

**风险点**: 见下方问题清单。

### 2.2 代码规模

| 统计项 | 数值 |
|--------|------|
| Python 源码（不含测试） | 28,082 行 / 55 模块 |
| 测试代码 | 38 文件 / 780+ 测试函数 |
| 总覆盖率 | **77.84%**（阈值 70%） |
| Skill 定义 | 14 个 SKILL.md |
| Agent 定义 | 14 个 .md |

## 三、发现的问题

### P0 — 严重

#### 3.1 Lock 文件与 CI Python 版本不一致

| 项目 | 详情 |
|------|------|
| **严重级别** | P0 |
| **证据** | `requirements.lock` 头部注释: `pip-compile with Python 3.14`；CI `ci-test.yml`: `python-version: '3.11'` |
| **影响** | Lock 文件在 3.14 环境下解析生成，CI 在 3.11 上安装。当依赖有 Python 版本条件标记时（如 `typing-extensions` 的版本区间），可能安装不兼容的版本组合，导致 CI 通过但实际运行失败，或反之 |
| **建议** | Lock 文件用 CI 一致的 Python 版本编译：`pip-compile --python-version 3.11`，或将 CI 升级到 3.14。统一二者即可 |

### P1 — 高

#### 3.2 七个超大模块（>1000 行），上帝对象风险

| 模块 | 行数 |
|------|------|
| `index_manager.py` | 2,029 |
| `state_manager.py` | 1,720 |
| `extract_chapter_context.py` | 1,702 |
| `rag_adapter.py` | 1,612 |
| `context_manager.py` | 1,312 |
| `status_reporter.py` | 1,244 |
| `anti_ai_scanner.py` | 1,056 |

| 项目 | 详情 |
|------|------|
| **严重级别** | P1 |
| **证据** | `wc -l` 统计，7 个文件均超 1000 行 |
| **影响** | 阅读困难、改动风险高、测试覆盖难做精细。`index_manager.py` 已经通过 mixin 拆分了部分逻辑（`index_entity_mixin` 995 行、`index_reading_mixin` 871 行、`index_debt_mixin` 504 行），但主体仍有 2029 行。`status_reporter.py` 覆盖率仅 36%，是最典型的上帝对象 |
| **建议** | 优先拆分 `status_reporter.py`（36% 覆盖率 + 1244 行）和 `extract_chapter_context.py`（73% 覆盖率 + 1702 行）。不急于拆分已有 mixin 体系的 `index_manager`，它的架构方向已经对了 |

#### 3.3 六个模块覆盖率远低于阈值，存在质量盲区

| 模块 | 覆盖率 | 行数 | 风险 |
|------|--------|------|------|
| `migration_auditor.py` | **0%** | 461 | 迁移逻辑完全无测试保护 |
| `ink.py`（入口） | **0%** | 12 | 入口虽小但 0% 属于疏漏 |
| `archive_manager.py` | **25%** | 567 | 归档操作涉及文件写入 |
| `runtime_compat.py` | **28%** | 78 | Windows 兼容层，非 Windows 环境天然低 |
| `status_reporter.py` | **36%** | 1,244 | 报告输出逻辑大量未覆盖 |
| `init_project.py` | **60%** | 885 | 项目初始化中 40% 路径未测试 |
| `security_utils.py` | **69%** | 589 | 安全工具低于 70% 阈值 |

| 项目 | 详情 |
|------|------|
| **严重级别** | P1 |
| **证据** | `pytest --cov` 输出 |
| **影响** | 整体 77.84% 通过了 70% 阈值，但这些模块的低覆盖率被其他高覆盖率模块稀释。`security_utils` 69% 差 1% 就跌破阈值——一旦其他模块微降，CI 可能意外通过而安全代码实际没有保护。`migration_auditor` 0% 意味着 `ink-migrate` 的核心逻辑完全是黑盒 |
| **建议** | 按影响排序补测试：security_utils（安全敏感）→ migration_auditor（迁移可靠性）→ status_reporter → archive_manager → init_project。`runtime_compat` 可标注 `# pragma: no cover`（Windows-only 分支），避免拖累指标 |

#### 3.4 Dashboard 零测试覆盖

| 项目 | 详情 |
|------|------|
| **严重级别** | P1 |
| **证据** | `ink-writer/dashboard/frontend/package.json` 无 test 脚本；无 jest/vitest 配置；`app.py` (531 行) 不在 pytest 覆盖范围内（.coveragerc `source = ink-writer/scripts`，不含 `dashboard/`） |
| **影响** | Dashboard 后端（FastAPI 531 行）和前端（React）完全无测试。虽然 Dashboard 定位为只读，但 `path_guard.py` 防路径穿越是安全关键逻辑，仅靠 CI 的 `import` 检查不够 |
| **建议** | 优先为 `path_guard.py` 加 pytest 单元测试（安全边界）。Dashboard API 层加基础冒烟测试。前端测试优先级最低，可以暂缓 |

### P2 — 中

#### 3.5 CI 触发路径不覆盖 Agent/Skill Markdown 变更

| 项目 | 详情 |
|------|------|
| **严重级别** | P2 |
| **证据** | `ci-test.yml` 的 `paths` 仅包含 `ink-writer/scripts/**`、`ink-writer/dashboard/**` 和配置文件。Agent (.md) 和 Skill (.md) 的修改不会触发 CI |
| **影响** | Agent prompt 修改（如改变 writer-agent 的约束规则）不经过任何自动化验证就合入 master。对于"约束即代码"的 Harness-First 架构，这等于核心逻辑变更绕过了 CI |
| **建议** | 考虑为 Agent/Skill 变更添加校验步骤（至少做 YAML front-matter 格式检查或 schema 验证），或在 CI 中运行一次 `ink-audit` 的轻量版 |

#### 3.6 `data_modules/ink.py` CLI 路由器 50% 覆盖率且职责过重

| 项目 | 详情 |
|------|------|
| **严重级别** | P2 |
| **证据** | `data_modules/ink.py` 498 行，覆盖率 50%。负责命令路由、参数解析、项目定位、子进程调用。160 行完全未覆盖（行 60-488 大量 Miss） |
| **影响** | 作为 Skill 到 Python 的唯一桥梁，半数代码路径未测试意味着新增命令或修改路由时可能引入回归 |
| **建议** | 拆分路由注册与执行逻辑。路由表单独维护，执行逻辑每个命令一个函数，方便独立测试 |

#### 3.7 `runtime_compat.py` 被 19 个模块导入但覆盖率仅 28%

| 项目 | 详情 |
|------|------|
| **严重级别** | P2 |
| **证据** | `grep -c "from runtime_compat"` 结果：19 个 .py 文件导入此模块。覆盖率 28%（78 行中仅 22 行被执行） |
| **影响** | 所有在 macOS/Linux 上运行的测试都走 early-return 路径，Windows 分支从未被验证。如果项目未来真的有 Windows 用户，这些代码是完全的盲区 |
| **建议** | 对 Windows 分支添加 `# pragma: no cover` 注释，或用 `unittest.mock` patch `sys.platform` 来测试 Windows 路径。前者更务实 |

## 五、总结

### 5.1 总体评价：✅ 部分合理

架构设计合理、核心链路可靠、工程纪律有基础。但存在锁文件版本不一致的隐患，以及多个模块的测试盲区可能在迁移、安全、报告等场景引发问题。Dashboard 作为新增组件缺乏测试覆盖也需要关注。

### 5.2 最关键的 3 个问题

| 排名 | 问题 | 原因 |
|------|------|------|
| 🥇 | Lock 文件 Python 3.14 vs CI Python 3.11 | 依赖解析不一致可能导致隐性兼容问题，且对所有模块生效 |
| 🥈 | 6 个模块覆盖率远低于阈值 | security_utils 69% + migration_auditor 0% = 安全和迁移两条关键路径裸奔 |
| 🥉 | Dashboard 零测试 + 不纳入覆盖率 | path_guard 是安全边界，531 行 FastAPI 无任何验证 |

### 5.3 最值得做的 3 个优化

| 排名 | 优化 | 预期收益 |
|------|------|----------|
| 🥇 | 统一 Lock 文件编译环境与 CI 版本 | 一条命令修复，消除最大隐患 |
| 🥈 | 补 security_utils + migration_auditor + path_guard 测试 | 3 个文件 ≈ 1250 行，安全 + 迁移路径从盲区变为可控 |
| 🥉 | 拆分 status_reporter.py（1244 行 / 36% 覆盖） | 消除最大的上帝对象，让覆盖率自然提升 |

### 5.4 先别乱动的地方

| 模块 | 原因 |
|------|------|
| `index_manager.py` + mixin 体系 | 已经在合理拆分中，2029 行虽大但 80% 覆盖率，mixin 架构方向正确，不要贸然重构 |
| Agent/Skill Markdown 定义 | 14 个 Agent + 14 个 Skill 是创作链路的核心 prompt，修改需配合实际写作验证，不是纯工程问题 |
| `rag_adapter.py` | 1612 行但 85% 覆盖率，向量检索逻辑特殊性强，拆分收益低风险高 |
| `extract_chapter_context.py` | 1702 行但 73% 覆盖率，属于核心数据管道，当前稳定运行中。可以排在 status_reporter 之后再考虑拆分 |
| 发布流水线 | 三个 workflow 配合良好，版本同步机制完善，除了修 Python 版本不一致外不需要动 |
