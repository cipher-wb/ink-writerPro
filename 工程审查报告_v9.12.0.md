## 一、项目目标与成功标准

### 项目目标

Ink Writer Pro 是一个**长篇网文工业化创作系统**，核心目标：

1. **自动化创作流水线** — 一条指令 `/ink-auto N` 完成 N 章写作 + 审查 + 修复 + 数据回写
2. **跨会话记忆一致性** — 25 张 SQLite 表 + RAG 向量检索，解决 LLM 长文遗忘问题
3. **可控质量保障** — 14 个 AI Agent + 计算型闸门 + 分层检查点，防幻觉/防崩坏/去 AI 味
4. **多平台部署** — Claude Code（主力）、Gemini CLI、Codex CLI 三平台支持

## 二、架构评估

### 2.1 Harness-First 架构（合理）

项目采用 **"约束驱动" 而非 "模型驱动"** 的架构思路，这是正确的工程决策：

- **计算型闸门 (Step 2C)** — 在昂贵的 LLM 审查前做确定性检查（字数、命名、伏笔逾期）
- **Reader Agent 升格** — 每章必跑 7 维评分，总分 ≥32 通过
- **分层检查点** — 5 章基础审查 / 10 章数据审计 / 20 章宏观审查

这个分层设计在"AI 生成 + 工程约束"的场景下是合理且必要的。

### 2.2 五层分层架构（基本合理）

```
Layer 5: CLI/Skills (ink.py, 13 个 skills)
Layer 4: Orchestrators (workflow_manager, context_manager)
Layer 3: Services (index_manager, state_manager, rag_adapter)
Layer 2: Config/Schema (config.py, schemas.py, state_schema.py)
Layer 1: Foundation (security_utils, project_locator, chapter_paths, runtime_compat)
```

**依赖方向基本正确**：高层依赖低层，无显著的反向依赖。

### 2.3 数据链设计（合理）

```
state.json ←→ StateManager ←→ SQLStateManager ←→ index.db
                                                      ↕
vectors.db ←→ RAGAdapter ←→ EmbeddingAPIClient
```

三个数据源各有职责，DataAgent 负责统一回写，ink-audit 负责一致性校验。

## 三、发现的结构性问题

### P1-1: SQLite 连接管理不一致 — context_manager.py 裸连接未关闭

**严重级别**: P1（高）  
**证据**:
- `context_manager.py:310` — `conn = sqlite3.connect(str(db_path))`，使用 try/finally 在 `line 411` 关闭
- 但中间有大量 try/except pass 块（line 316-409），如果在 `conn = sqlite3.connect()` 和 `try:` 之间发生异常，连接泄漏
- 对比：`index_manager.py:925-934` 和 `rag_adapter.py:254-261` 和 `style_sampler.py:87-95` 已正确使用 `@contextmanager` 的 `_get_conn()` 模式

**影响**:
- Windows 平台文件锁死（SQLite WAL 文件无法释放）
- 长运行 ink-auto 批量写作场景下文件描述符耗尽
- conftest.py 中的 `ignore:unclosed database:ResourceWarning` 掩盖了真实问题

**建议**:
- context_manager.py 应仿照 index_manager 的 `_get_conn()` 模式重构
- 移除 conftest.py 中的 ResourceWarning 过滤器，暴露真实泄漏

---

### P1-2: 测试 ResourceWarning 过滤器掩盖真实缺陷

**严重级别**: P1（高）  
**证据**:
- `conftest.py:22` — `"ignore:unclosed database:ResourceWarning"` 全局压制
- 上游的 `index_manager`、`rag_adapter`、`style_sampler` 已修复为 `_get_conn()` 模式
- 但 `context_manager.py:310` 仍有裸连接，该过滤器使其静默通过测试

**影响**:
- 新引入的连接泄漏无法在测试阶段被发现
- CI 绿灯不代表无资源泄漏

**建议**:
- 修复 context_manager.py 后，移除该过滤器
- 改为在测试中使用 `pytest.warns(ResourceWarning)` 显式断言无泄漏

### P2-1: 无 pyproject.toml — 依赖管理分散

**严重级别**: P2（中）  
**证据**:
- 项目使用 3 个 `requirements.txt` + 2 个 `requirements.lock` 管理依赖
- 无 `pyproject.toml`、`setup.py`、`setup.cfg`
- 测试依赖（pytest, pytest-cov）混在核心依赖 `requirements.txt` 中，未分组

**影响**:
- 无法用 `pip install -e .` 做可编辑安装
- 测试依赖与运行时依赖混合，生产环境可能安装不必要的包
- 缺少标准的包元数据（requires-python、classifiers 等）

**建议**:
- 考虑引入 `pyproject.toml` 统一管理（不急迫，当前 requirements.txt 体系能用）
- 至少将测试依赖分组（如 `requirements-dev.txt`）

### P3-1: 仅 2 个模块定义了 `__all__`

**严重级别**: P3（低）  
**证据**:
- 仅 `scripts/__init__.py` 和 `data_modules/__init__.py` 定义了 `__all__`
- 其余 ~50 个子模块均无 `__all__` 定义

**影响**:
- IDE 自动补全会暴露内部实现细节
- `from module import *` 行为不可预测
- 对外部使用者（如 Skills 中的 Python 调用）不友好

**建议**:
- 低优先级，当前项目主要通过 `data_modules/__init__.py` 的 59 个显式导出控制 API
- 仅在子模块被外部直接 import 时才需要加

---

### P3-2: requirements.lock 基于 Python 3.14 编译

**严重级别**: P3（低）  
**证据**:
- `scripts/requirements.lock` 和 `dashboard/requirements.lock` 均由 Python 3.14 的 pip-compile 生成
- Python 3.14 目前处于 beta 阶段（预计 2026-10 正式发布）

**影响**:
- 依赖解析结果可能与 3.12/3.13 不同（某些包的 wheel 可用性不同）
- 如果用户本地使用 3.12，lock 文件可能含有 3.14-only 的包版本

**建议**:
- CI 已固定 Python 3.14 运行，当前不影响 CI
- 若需支持更低版本，考虑用 3.12 重新编译 lock 文件
