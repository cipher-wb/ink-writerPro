**审查日期**: 2026-04-03  
**审查范围**: 项目级全量审查（架构、模块边界、配置、依赖、测试、发布链路）

## 一、项目目标与成功标准

**项目定位**: 工业级中文网文自动化写作系统，通过 14 个 AI Agent 协同，实现"一键写作、自动审查、自动修复、自动规划"的全流程自动化。

**核心成功标准**（来源：README.md）：
- 在检查点之间无需人工干预即可生成可发布内容
- 通过 25 张 SQLite 表维护 300+ 章的角色一致性
- 通过 6 层反检测 + 200 词禁词表通过 AI 检测
- 7 维读者参与度评分达标（Reader Simulator Agent）
- 关键/高优问题在质量检查点内自动修复

## 二、架构评估

### 当前架构

```
Skill 层 (14 个入口命令)
    ↓
Agent 层 (4 核心 + 10 审查 Agent，Markdown 定义)
    ↓
CLI/脚本层 (ink.py 统一入口)
    ↓
领域层 (StateManager, SQLStateManager, EntityLinker, ContextManager)
    ↓
数据访问层 (IndexManager + 5 Mixin, RAGAdapter, SnapshotManager)
    ↓
存储层 (SQLite: index.db, JSON: state.json, Vector: vectors.db)
```

### 架构判断

**架构整体支撑目标**。具体评估：

| 维度 | 评估 | 说明 |
|------|------|------|
| 跨会话隔离 | ✅ 优秀 | 每章独立 CLI 会话，自动上下文清理 |
| 分层检查点 | ✅ 优秀 | 5/10/20 章递进检查，质量逐级升级 |
| 多 Agent 共识 | ✅ 优秀 | 10 独立审查视角，非单一评审 |
| 延迟加载 | ✅ 优秀 | `importlib` 懒加载防循环依赖 |
| Mixin 组合 | ✅ 良好 | IndexManager 5 个 Mixin 职责清晰 |
| 双存储策略 | ✅ 良好 | JSON 轻量元数据 + SQLite 大数据，各司其职 |

## 三、发现的结构性问题

### 问题 1：前端 package-lock.json 版本滞后

| 项目 | 内容 |
|------|------|
| **严重级别** | 🟡 中等 |
| **证据** | `ink-writer/dashboard/frontend/package-lock.json` 版本为 **9.5.0**，而 `package.json` 及其余 6 个版本文件均为 **9.7.0** |
| **影响** | `npm ci` 将安装旧版依赖；`sync_plugin_version.py --check` 未覆盖此文件，版本校验存在盲区 |
| **建议** | 1) 在 `dashboard/frontend/` 下执行 `npm install` 刷新 lock 文件；2) 将 package-lock.json 纳入 `sync_plugin_version.py` 的校验范围 |

### 问题 2：发布流水线未集成前端构建

| 项目 | 内容 |
|------|------|
| **严重级别** | 🟡 中等 |
| **证据** | `.github/workflows/plugin-release.yml` 仅做版本校验 + git tag + GitHub Release，**无 `npm ci && npm run build` 步骤** |
| **影响** | Dashboard 前端产物需手动构建并提交，发布可能携带过期 UI |
| **建议** | 在 release workflow 的 tag 创建前增加前端构建步骤，或建立独立的前端构建 workflow |

### 问题 3：Lock 文件 Python 版本与 CI 不一致

| 项目 | 内容 |
|------|------|
| **严重级别** | 🟢 低 |
| **证据** | `requirements.lock` 文件头标注由 **Python 3.14** 的 pip-compile 生成，而 `.github/workflows/ci-test.yml` 使用 **Python 3.11** |
| **影响** | 当前依赖前向兼容未出问题，但版本差异可能在未来引入隐患 |
| **建议** | 用 Python 3.11 重新生成 lock 文件，或将 CI 升级至与开发环境一致的版本 |

### 问题 4：约半数模块缺乏单元测试

| 项目 | 内容 |
|------|------|
| **严重级别** | 🟡 中等 |
| **证据** | 34 个核心模块中 **16 个缺少专属测试文件**（47%），包括：5 个 index mixin、ink.py (主 CLI)、snapshot_manager.py、style_sampler.py、query_router.py、genre_profile_builder.py、writing_guidance_builder.py 等 |
| **影响** | 覆盖率虽达 70% 阈值（主要由 index_manager 测试间接覆盖 mixin），但 CLI 入口和写作指导构建器等关键模块无直接测试保护 |
| **建议** | 优先补充：1) `snapshot_manager.py`（频繁在上下文组装中使用）；2) `style_sampler.py`（影响文风一致性）；3) `writing_guidance_builder.py`（影响写作指导输出质量） |

### 问题 5：Dashboard 完全无测试

| 项目 | 内容 |
|------|------|
| **严重级别** | 🟢 低 |
| **证据** | `ink-writer/dashboard/` 下的 Python 后端（server.py、app.py、watcher.py）和前端（React/Vite）均无测试文件 |
| **影响** | Dashboard 为只读可视化工具，风险较低，但 `path_guard.py` 涉及安全（路径遍历防护），值得测试覆盖 |
| **建议** | 至少为 `path_guard.py` 和 `server.py` 的 API 端点补充单元测试 |

### 问题 6：`project_locator.py` 过度使用宽泛异常捕获

| 项目 | 内容 |
|------|------|
| **严重级别** | 🟢 低 |
| **证据** | `project_locator.py` 中有 **7 处** `except Exception as e:` 宽泛捕获（第 57、87、110、133、224、239、347 行） |
| **影响** | 可能掩盖特定错误类型（如 PermissionError vs FileNotFoundError），增加调试难度 |
| **建议** | 收窄为具体异常类型（OSError、ValueError、json.JSONDecodeError） |

### 问题 7：SQL Schema 操作使用 f-string

| 项目 | 内容 |
|------|------|
| **严重级别** | 🟢 低 |
| **证据** | `index_manager.py` 第 939、947 行使用 f-string 拼接 PRAGMA 和 ALTER TABLE 语句 |
| **影响** | 实际风险极低（表名/列名均为代码内部硬编码常量），但不符合安全编码最佳实践 |
| **建议** | 维持现状即可；若要改进，可对 table_name/column_name 增加白名单校验 |

## 四、优势亮点

| 维度 | 表现 |
|------|------|
| **模块边界** | 零循环依赖，lazy import + mixin 组合模式干净 |
| **版本管理** | `sync_plugin_version.py` 集中管理 6 个版本文件 + CI 校验 |
| **安全防护** | `path_guard.py` 防路径遍历、`security_utils.py` 防命令注入/文件名注入、原子写入 |
| **配置集中** | 所有魔法数字集中在 `config.py`，有文档说明意图 |
| **无死代码** | 34 个模块均有明确用途，`__all__` 导出规范 |
| **错误处理** | 无裸 `except:`，全局使用 `logging.getLogger(__name__)` |

## 六、最关键的 3 个问题

| 排名 | 问题 | 原因 |
|------|------|------|
| 1 | **前端 package-lock.json 版本滞后** | 版本同步工具的唯一盲区，可能导致发布携带旧依赖 |
| 2 | **发布流水线未集成前端构建** | 手动构建是人为失误的温床 |
| 3 | **47% 核心模块缺少专属测试** | 虽达 70% 阈值但间接覆盖脆弱，重构时缺乏安全网 |

---

## 七、最值得做的 3 个优化

| 排名 | 优化 | 预期收益 |
|------|------|----------|
| 1 | **将 package-lock.json 纳入版本同步工具** | 一次性解决，彻底消除版本盲区 |
| 2 | **release workflow 增加前端构建步骤** | 自动化 > 人工，防止发布事故 |
| 3 | **为 snapshot_manager / style_sampler / writing_guidance_builder 补测试** | 这 3 个模块在写作核心路径上频繁调用，投入产出比最高 |

---

## 八、先别乱动的地方

| 区域 | 原因 |
|------|------|
| **IndexManager + 5 Mixin 架构** | 当前 mixin 组合模式工作良好，拆分反而增加复杂度 |
| **`__init__.py` 的 lazy import 机制** | 这是防循环依赖的关键基础设施，动了可能牵一发而动全身 |
| **config.py 的魔法数字** | 虽然数值多，但都有文档注释且集中管理，抽象为配置文件收益不大 |
| **project_locator.py 的宽泛异常** | 低优先级，该模块是探测性质的代码（找项目根目录），宽泛捕获反而是合理策略 |
| **SQL f-string** | 实际无安全风险（硬编码常量），改动引入的回归风险大于收益 |
