## 三、发现的问题

### P0 — 严重

#### 3.1 测试泄漏 MagicMock 文件到项目根目录（新发现）

| 项目 | 详情 |
|------|------|
| **严重级别** | P0 |
| **证据** | 项目根目录存在 50 个名为 `<MagicMock name='mock.DataModulesConfig.from_project_root().index_db' id='...'>` 的文件，每个 396KB（SQLite 数据库），合计约 20MB。每次 `pytest` 运行增加 2 个 |
| **泄漏源** | `test_backup_manager.py:264` — `test_backup_index_db_exception_is_swallowed` |
| **根因分析** | 测试使用 `patch.dict("sys.modules", {"data_modules.config": MagicMock(side_effect=ImportError(...))})` 意图模拟导入失败。但 `side_effect` 仅在 MagicMock 被**调用**时触发，不阻止**属性访问**。因此 `from data_modules.config import DataModulesConfig` 成功获取一个 MagicMock 子属性（而非抛出 ImportError）。后续代码通过该 MagicMock 获取 `index_db` 和 `ink_dir` 路径（均为 MagicMock 的 `__str__` 表示），真实的 `IndexManager`（未被 mock）在这些路径上创建了 SQLite 数据库 |
| **影响** | (1) 磁盘空间持续膨胀（每次 CI/本地测试 +800KB）(2) 虽已被 `.gitignore` 排除不会进入仓库，但污染本地开发环境 (3) 暴露了 mock 隔离不完整的系统性风险——如果 `.gitignore` 没有兜底，这些文件会被 `git add .` 提交 |
| **建议** | (1) 修复 mock：需要同时 patch `data_modules.index_manager`，或使用 `patch("data_modules.config.DataModulesConfig")` 而非 `patch.dict("sys.modules", ...)`。(2) 清理已泄漏的文件。(3) 在 `conftest.py` 中添加全局 fixture 验证测试不在项目根目录创建文件 |

## 五、总结

### 5.1 总体评价：✅ 部分合理（趋势向好）

与 v9.9.0 相比，核心工程问题修复进展显著（P0 已解决、security/migration 覆盖率大幅提升）。但新发现的 MagicMock 泄漏是 P0 级别问题，暴露了 mock 隔离的系统性风险。Dashboard 和 status_reporter 的测试盲区仍需填补。

### 5.2 最关键的 3 个问题

| 排名 | 问题 | 原因 |
|------|------|------|
| 1 | MagicMock 文件泄漏到项目根目录 | mock 隔离不完整，每次测试泄漏 2 个 SQLite 文件，已累积 50 个/20MB |
| 2 | Dashboard 后端 395 行零测试 | app.py 289 行 FastAPI 逻辑完全无保护 |
| 3 | status_reporter 36% 覆盖率连续两版未改善 | 589 行中 377 行未覆盖，是代码量 x 未覆盖率的最大盲区 |

### 5.3 最值得做的 3 个优化

| 排名 | 优化 | 预期收益 |
|------|------|----------|
| 1 | 修复 `test_backup_index_db_exception_is_swallowed` mock + 清理泄漏文件 | 消除 P0，5 分钟修复 |
| 2 | `plugin-version.yml` Python 版本对齐 3.14 | 一行改动，消除最后的 CI 版本不一致 |
| 3 | Dashboard `app.py` 加 10 个冒烟测试 | 289 行从 0% 到约 50%，整体覆盖率可提升 ~1% |

### 5.4 先别乱动的地方

| 模块 | 原因 |
|------|------|
| `index_manager.py` + mixin 体系 | 82% 覆盖率，mixin 架构方向正确，不要贸然重构 |
| Agent/Skill Markdown 定义 | 13 Skill + 14 Agent 是创作链路的核心 prompt，修改需配合实际写作验证 |
| `rag_adapter.py` | 1612 行但 85% 覆盖率，向量检索逻辑特殊性强 |
| `extract_chapter_context.py` | 1702 行但 73% 覆盖率，核心数据管道，当前稳定 |
| 发布流水线（除版本号修复外） | 三个 workflow 配合良好，版本同步机制完善 |
| `state_manager.py` | 1720 行但 90% 覆盖率，高覆盖 = 高稳定 |
