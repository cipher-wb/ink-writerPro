# 工程审查报告 — ink-writer v9.14.0

> 审查日期：2026-04-04
> 审查范围：项目架构、模块边界、依赖管理、测试体系、CI/CD 发布链路、配置管理
> 目标平台：macOS

---

## 一、项目目标与成功标准

**项目定位**：基于 Claude Code / Gemini CLI / Codex CLI 的长篇网文自动化创作系统，通过 14 个 AI Agent + 25 张 SQLite 表 + RAG 向量检索，实现"一条指令自动写书、审查、修复、规划"。

**成功标准**（从 README 和架构推断）：

| 维度 | 标准 |
|------|------|
| 功能完整性 | 支持 init → plan → write → review → polish → data-extract 全流程 |
| 数据一致性 | state.json + index.db 双存储同步，实体/关系/时间线零丢失 |
| 质量把控 | 10 个 Checker Agent 并行审查，70% 以上测试覆盖率 |
| 可发布性 | plugin.json 版本统一，CI 绿灯，一键 GitHub Release |
| 跨平台 | 同时支持 Claude Code / Gemini / Codex 三端 |

## 二、架构评估

### 当前架构概览

```
skills/ (13 个 Skill 入口)
  ↓ (调用 Python 数据链)
agents/ (14 个 Agent 定义，.md prompt)
  ↓ (通过 Skill 编排)
scripts/data_modules/ (核心 Python 模块，~27K LOC)
  ├── config.py (配置单例)
  ├── index_manager.py (2069 LOC，5 个 Mixin)
  ├── state_manager.py (1721 LOC)
  ├── context_manager.py (1318 LOC)
  ├── rag_adapter.py (1613 LOC)
  ├── entity_linker.py (274 LOC)
  └── ... (20+ 模块)
dashboard/ (FastAPI + Vue.js，只读 Web UI)
```

**架构判断：总体合理。** 模块分层清晰、依赖方向单一、无循环导入。lazy import + mixin 分拆有效控制了复杂度。但存在以下结构性问题需要关注。

## 三、发现的问题

### P1 — 严重

#### 问题 1：requirements.txt 声明 Python ≥ 3.10，实际最低要求为 3.12

| 项 | 内容 |
|----|------|
| **严重级别** | P1 — 用户可能在 3.10/3.11 上安装后遇到运行时错误 |
| **证据** | `ink-writer/scripts/requirements.txt:2` 声明 `# Python >= 3.10 required`；CI 矩阵（`.github/workflows/ci-test.yml:34`）仅测试 `['3.12', '3.14']`；最近 commit 信息明确写"最低版本声明修正为 3.12" |
| **影响** | 3.10/3.11 用户按文档安装后可能遇到语法或标准库兼容错误，排查成本高 |
| **建议** | 将 `requirements.txt` 第 2 行改为 `# Python >= 3.12 required`，与 CI 矩阵和 commit 记录对齐 |

#### 问题 2：Lock 文件仅锁定 Python 3.14，CI 还跑 3.12 矩阵

| 项 | 内容 |
|----|------|
| **严重级别** | P1 — 3.12 CI 用锁定的 3.14 依赖可能安装失败 |
| **证据** | `ink-writer/scripts/requirements.lock:2` 注释 `autogenerated by pip-compile with Python 3.14`；`ink-writer/dashboard/requirements.lock:2` 同理；CI 矩阵包含 `3.12` |
| **影响** | 若某个传递依赖发布了 3.14-only wheel（例如 `uvloop==0.22.1` 的 build tag），3.12 矩阵会 pip install 失败。当前碰巧能跑不代表下次更新 lock 后还能跑 |
| **建议** | 为 3.12 单独生成一份 lock 文件（`requirements-3.12.lock`），或在 CI 中对 3.12 使用 `requirements.txt`（非 lock）安装 |

---

### P2 — 中等

#### 问题 3：测试依赖与生产依赖混写在同一 requirements.txt

| 项 | 内容 |
|----|------|
| **严重级别** | P2 — 生产部署时多装 pytest/pytest-cov 等测试包 |
| **证据** | `ink-writer/scripts/requirements.txt` 在同一文件中列出 `aiohttp` 和 `pytest>=7.0.0`、`pytest-cov>=4.1.0` 等 |
| **影响** | Dashboard 生产部署会装入不必要的测试依赖，增加攻击面和安装时间 |
| **建议** | 拆分为 `requirements.txt`（生产）+ `requirements-dev.txt`（测试），CI 安装 dev，生产安装 prod |

#### 问题 4：CI 仅在 ubuntu-latest 运行，用户主平台为 macOS

| 项 | 内容 |
|----|------|
| **严重级别** | P2 — macOS 特有问题无法在 CI 中暴露 |
| **证据** | `.github/workflows/ci-test.yml:31` 固定 `runs-on: ubuntu-latest`；项目 README 安装示例面向 macOS 用户 |
| **影响** | 文件路径大小写敏感性差异（macOS 默认 case-insensitive）、SQLite 文件锁行为差异、`uvloop` 在 macOS 上的编译问题等无法在 CI 中捕获 |
| **建议** | 在 CI 矩阵中增加 `macos-latest`，至少覆盖主分支的 push 事件 |

#### 问题 5：IndexManager 成为中心枢纽，6+ 模块直接依赖

| 项 | 内容 |
|----|------|
| **严重级别** | P2 — 改动 IndexManager API 波及面过大 |
| **证据** | 以下模块顶层 import IndexManager：`sql_state_manager.py:23`、`rag_adapter.py:33`、`context_manager.py:25`、`entity_linker.py:21`；`state_manager.py` 在方法内部延迟 import（:473）；`style_sampler.py` 条件 import（:464）。`context_manager.py` 中对 IndexManager 的方法调用超过 40 处 |
| **影响** | IndexManager 的表结构变更或查询方法签名调整需要同步修改 6 个以上消费模块，回归测试范围大 |
| **建议** | 短期：给 IndexManager 的公开方法加稳定性标注（stable / experimental）。中期：抽取查询接口（Protocol / ABC），让消费者依赖接口而非实现 |

#### 问题 6：三层状态同步链异常被静默吞掉

| 项 | 内容 |
|----|------|
| **严重级别** | P2 — 数据丢失风险 |
| **证据** | `state_manager.py:467` 附近：`except Exception as exc: logger.warning("SQLite sync failed ..."); return False`。SQLite 同步失败仅打 warning，调用方无法区分"同步成功"和"同步失败但数据只在 JSON 中" |
| **影响** | state.json 与 index.db 静默不一致，下游 context_manager / rag_adapter 查询到过期数据，导致上下文包质量下降。用户无感知 |
| **建议** | 将异常分类：可恢复的（如临时锁冲突）保留 warning + 重试；不可恢复的（schema 不匹配等）升级为 error 并标记同步状态为 `stale`，供 ink-audit 检测 |

---

### P3 — 低

#### 问题 7：无 pyproject.toml，缺乏现代 Python 包管理

| 项 | 内容 |
|----|------|
| **严重级别** | P3 — 不影响运行，但影响工程规范性 |
| **证据** | 项目根目录和 ink-writer/ 下均无 `pyproject.toml`。版本号分散在 `plugin.json`、`marketplace.json`、`scripts/__init__.py` 三处，依赖 `sync_plugin_version.py` 脚本同步 |
| **影响** | 无法 `pip install -e .`，IDE 对模块解析不友好，版本管理需额外脚本 |
| **建议** | 当前 `sync_plugin_version.py` 机制可工作，暂不急。若未来需要 pip 分发，再迁移到 pyproject.toml |

#### 问题 8：6 个源码模块缺少测试覆盖

| 项 | 内容 |
|----|------|
| **严重级别** | P3 — 多数为入口薄包装或兼容层 |
| **证据** | 以下模块在 `data_modules/tests/` 中无对应测试文件：`ink.py`（CLI 入口）、`chapter_outline_loader.py`、`quality_trend_report.py`、`update_state.py`、`runtime_compat.py`、`scripts/__init__.py` |
| **影响** | CLI 入口 `ink.py` 缺少冒烟测试，命令分发路由变更可能引入回归 |
| **建议** | 优先为 `ink.py` 添加 CLI 冒烟测试（验证子命令分发）；`runtime_compat.py` 几乎全是 `# pragma: no cover` Windows 代码，macOS 平台不需要 |

#### 问题 9：残留 Windows 专属文件

| 项 | 内容 |
|----|------|
| **严重级别** | P3 — 对 macOS 用户无害但造成困惑 |
| **证据** | `ink-writer/scripts/run_tests.ps1`（PowerShell 脚本）；`runtime_compat.py` 中全部逻辑均为 Windows 路径处理（:16-77），macOS 上直接返回 `Path(value)` |
| **影响** | 无功能影响，但增加维护认知负荷 |
| **建议** | 如果明确只支持 macOS，可在文档中标注 Windows 支持为 best-effort，不需要删除这些文件 |

## 四、做得好的地方（不需要动的）

| 做法 | 证据 |
|------|------|
| **Lazy import 防循环依赖** | `data_modules/__init__.py:103-111` 使用 `__getattr__` 延迟加载 59 个导出 |
| **Mixin 分拆大类** | IndexManager 通过 5 个 Mixin 将 2069 LOC 分布到逻辑子模块 |
| **依赖方向单一** | 全量 grep 确认无循环 import：消费者 → IndexManager → config/observability |
| **文件锁保护状态写入** | `state_manager.py` 使用 `filelock` 防止并发写 state.json |
| **Dashboard 只读设计** | `app.py` 只暴露 GET 接口 + path_guard 路径穿越防护 |
| **版本同步脚本** | `sync_plugin_version.py` + CI `plugin-version.yml` 确保三处版本号一致 |
| **Lock 文件可复现安装** | scripts 和 dashboard 各自有 `requirements.lock`，pip-compile 生成 |
| **测试覆盖率门禁** | pytest.ini 配置 `--cov-fail-under=70`，CI 强制执行 |

## 五、总结

### 1. 总体评价：合理

架构分层清晰，模块职责明确，依赖方向健康。对于一个以 Claude Code Skill/Agent 为核心的创作系统来说，当前的 Python 数据链设计是务实且可维护的。主要问题集中在工程配置层面而非架构设计。

### 2. 最关键的 3 个问题

| 排名 | 问题 | 原因 |
|------|------|------|
| **#1** | Lock 文件仅锁 3.14，CI 矩阵含 3.12 | 随时可能导致 CI 红灯，且当前"碰巧能跑"给人虚假安全感 |
| **#2** | Python 版本声明 ≥3.10 与实际 ≥3.12 不一致 | 直接误导用户，导致安装后运行时错误 |
| **#3** | 状态同步异常被静默吞掉 | 数据层面的隐性故障最难排查，可能导致写了几十章后才发现实体丢失 |

### 3. 最值得做的 3 个优化

| 排名 | 优化 | 投入/产出比 |
|------|------|------------|
| **#1** | 修正 requirements.txt 的 Python 版本声明为 ≥3.12 | 改一行注释，消除误导，5 分钟 |
| **#2** | CI 增加 `macos-latest` 矩阵 | 改 2 行 YAML，覆盖主用户平台，10 分钟 |
| **#3** | 拆分 requirements.txt 为 prod / dev | 小改动，消除生产环境安装测试依赖的问题，15 分钟 |

### 4. 先别乱动的地方

| 区域 | 理由 |
|------|------|
| **IndexManager 的 Mixin 架构** | 虽然是 hub 模块，但 5 个 Mixin 分拆已经是当前最佳折中。抽接口的收益需要等到有第二种存储后端时才显现，现在做是过度设计 |
| **state.json + index.db 双存储** | 这是历史迁移的中间态，但当前同步机制在正常路径下工作正确。贸然合并为纯 SQLite 会破坏已有项目的兼容性 |
| **三平台（Claude/Gemini/Codex）抽象** | 各平台通过独立的 references 文档（`claude-code-call-matrix.md`, `gemini-tools.md`, `codex-tools.md`）描述差异，而非代码层抽象。这对"以 prompt 为核心"的系统来说是正确做法 |
| **runtime_compat.py / run_tests.ps1** | Windows 兼容代码虽然 macOS 用不到，但不占运行时开销（短路返回），删除反而可能破坏其他贡献者的环境 |
| **没有 pyproject.toml** | 当前作为 Claude Code Plugin 分发，不需要 pip install。为加 pyproject.toml 而重构目录结构，投入产出比不划算 |
