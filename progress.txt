## Codebase Patterns
- Case dataclass 字段新增时必须同步：(1) `ink_writer/case_library/models.py` `Case.from_dict`/`to_dict`；(2) `schemas/case_schema.json` properties；(3) 现有 410 个 yaml 必须 backward-compatible（新字段 optional + from_dict 容错）。
- 新增测试目录必须加入 `pytest.ini` testpaths 单行（不会自动发现）；M5 已加 `tests/regression_tracker tests/meta_rule_emergence tests/dashboard tests/learn`。
- 所有 `open()` 必带 `encoding="utf-8"`（CLAUDE.md Windows 兼容守则）。
- Python 入口可执行脚本须调 `runtime_compat.enable_windows_utf8_stdio()`。
- 项目用 python3（不是 python）；testpaths 已有 ~50 个目录单行；fixtures 在 `tests/<module>/conftest.py`。
- legacy case yaml 路径：`data/case_library/cases/CASE-*.yaml`（410 个）。
- `__main__.py` Windows UTF-8 引导块（参考 `ink_writer/creativity/__main__.py`）：先 `import os/sys` 把 `ink-writer/scripts` 加到 `sys.path`，try/except 调 `runtime_compat.enable_windows_utf8_stdio()`，然后才能 import 业务模块。pyproject 根 `exclude` 已豁免 ruff E402/I001 风险（`tests*`/`ink-writer*` 都不入 ruff，`ink_writer/**/__main__.py` 同 pattern 已有先例可参照）。
- `pyproject.toml` 顶层 `exclude` 排除了 `tests*`，所以 ruff 只 lint `ink_writer/**`；新增模块只需保证主代码 `ruff check` 全绿。
- 严重程度阶梯（regression_tracker / record_recurrence 共用）：`P3 → P2 → P1 → P0`，到 P0 不再升；累计次数靠 `len(recurrence_history)`。常量 `_SEVERITY_LADDER` 在 `ink_writer/case_library/store.py`。
- evidence_chain 检查器命中 case_id 至少 4 种载体：`checkers[].cases_hit` / `checkers[].cases_violated` / `checkers[].details.cases_hit` / `checkers[].details.cases_violated`；既支持 `["CASE-..."]` 字符串，也支持 `[{"case_id": "CASE-..."}]` dict。M5 解析器须把这些分支都覆盖（见 `regression_tracker/tracker.py::_hit_case_ids`）。
- M5 元规则提议 ID 分配复用模式（`meta_rule_emergence/emerger.py::_next_proposal_id`）：扫目录 → regex `MR-(\d{4,})` → max+1 → 默认 `MR-0001`；缺目录直接返回 `MR-0001`。后续 US-004/US-005 列表/审批 CLI 可直接复用。
- LLM 客户端注入用 Protocol 类型 `class _LLMLike(Protocol)` + `messages.create(...)` 接口（见 `meta_rule_emergence/emerger.py`）；测试塞入 `_FakeLLM` 即可不依赖真 LLM；与 `scripts/corpus_chunking/llm_client.LLMClient` 同形。
- LLM 输出 JSON 解析容错（`_parse_llm_json`）：先 strip ```json ... ``` 围栏 → `json.loads` → 失败再 regex `\{.*\}` 抓最大块；M5 后续 LLM 调用都建议复用此模式。
- M5 子命令模块 (`meta_rule_cli` / 后续 dashboard/learn) 接入 `case_library/cli.py` 的标准模式：(1) 模块自带 `register_subparsers(parent_sub)` 注册嵌套子命令 + `dispatch(args)` 解 namespace；(2) cli.py 顶部 import + `_build_parser` 末尾调一次 `register_..._subparsers(sub)` + main() if 链多一行 dispatch 路由；(3) 自身仍保留独立 `_build_parser/main` 以便 `python -m ...` 直跑。这样新增子命令对 cli.py 的入侵 ≤ 4 行。
- meta-rule 提议 yaml 持久化字段：`status: pending|approved|rejected` + 仅在 approve/reject 时分别加 `approved_at`/`rejected_at`（YYYY-MM-DD UTC）；`covered_cases` 是 case_id 列表，approve 时遍历 store.load → 写 `case.meta_rule_id` → store.save。幂等：第二次 approve 看到 status != pending 直接 rc=1 + stderr 报错；不会重写 cases。
- dashboard 数据布局（M5 P3 起 aggregator 假定的 base_dir layout）：`<base_dir>/.dry_run_counter`（M3 章节）+ `<base_dir>/.planning_dry_run_counter`（M4 策划期）+ `<base_dir>/<book>/chapters/*.evidence.json`（chapter 级 outcome=delivered/needs_human_review/blocked）+ `<base_dir>/<book>/planning_evidence_chain.json`（含 stages list + overall_passed）+ `<base_dir>/case_library/meta_rules/MR-*.yaml`（status: pending|approved|rejected）+ `<base_dir>/editor_reviews/*.yaml`（编辑评分 trend）。M5 dashboard / 周报 / overview JSON 都按这一布局工作；测试用 `tmp_path / "data"` 现造同形布局即可零耦合验证。
- chapter evidence pass 判定：`outcome == "delivered"` = 通过；`needs_human_review` 与 `blocked` 都不计 pass。M3 周报 / 切换推荐 都用此契约。
- planning evidence pass 判定：每个 stage `outcome != "blocked"` = 通过（M4 与 §6.2 一致；spec 注 outcome=blocked → 切真阻断阻塞）。
