## Codebase Patterns
- 评估 evidence_chain 字段时优先扩展现有 dataclass（默认值保证向后兼容），不要新建并行 dataclass

- ink_writer/evidence_chain/__init__.py 是单一导出口，新增 API 同步在此导出 + 加到 __all__
- 测试 fixtures 走 tests/<area>/conftest.py，fixture 命名 `<area>_base_dir` + `sample_<area>_<scenario>`
- 阶段性输出（ink-init / ink-plan）合并到同一 JSON 时按 stage 名去重再追加，overall_passed 用 all() 聚合
- pytest.ini testpaths 是单行（极长但不能换行），新加测试目录追加到末尾
- 新 checker 完全照 M3 protagonist_agency 模式：`ink_writer/checkers/<name>/{__init__,models,checker}.py + prompts/check.txt`，dataclass 含 score/blocked/cases_hit/notes + to_dict()，cases_hit 默认 []（由编排层注入），LLM/JSON 失败统一降级为 score=0.0/blocked=True/notes 含 "checker_failed"
- 双层（regex + LLM）checker 模式：先用 `re.escape(kw)` 拼 OR pattern 字面扫描，命中直通不调 LLM；regex miss 才走 LLM 语义判断；report 同时暴露 `regex_match: bool` + `llm_match: bool|None`，让 evidence_chain 调试时一眼看出走的哪条路径
- 纯规则 checker（无 LLM）跳过 prompts/ 目录与 mock_llm_client；保守阻断契约：依赖文件缺失 → score=0.0 + blocked=True + notes 含 'missing'，让 planning_review 拿到失败信号
- LLM 响应统一用 _extract_json* helper：先 try fence ```json ... ```，再退化为 raw "{ ... }" / "[ ... ]" 子串解析；失败抛 ValueError 让 except 捕获重试
- 跨 checker 复用 tests/checkers/conftest.py 的 FakeLLMClient（queue + calls + .messages.create stub），不要在 sub-area 新建 fixture
- agent spec 落 ink-writer/agents/<name>-checker.md，frontmatter 含 name/description/tools=Read/model=inherit；包含来源 spec/PRD、Python 调用示例、字段表、阻断行为、与下游衔接段
- 数据爬虫脚本：scripts/<area>/__init__.py（空）+ fetch_*.py（UA 含联系邮箱、_check_robots_txt、RATE_LIMIT_SECONDS=1.0、_load_progress/_save_progress_one 续爬、_fetch_*_page → list[(rank, id)]、_fetch_one_* max_retries 失败 return None、enable_windows_utf8_stdio()）；测试用 autouse fixture monkeypatch OUTPUT_PATH/PROGRESS_PATH 到 tmp_path 隔离仓库 data/

- **Learnings for future iterations:**
  - 双层 checker 的 regex 扫描必须**只在前 N 章 summary** 上跑，不要扩到全 outline（spec §3.5 明确 "前 3 章"）；helper `_first_n_chapters(skeleton[:3])` 是单一真相源
  - `re.escape(kw)` 拼 `|` OR pattern 是关键：金手指关键词可能含正则元字符（如 `[]` `()`），不 escape 会炸 `re.error`；编译一次 pattern 后顺序扫 entry 取首个命中的 `chapter_idx`
  - LLM 返回 `matched_chapter` 必须做范围校验 `1 <= mc <= 3`，否则 LLM 偶尔会返回 `4` 或 `null` 让 `report.matched_chapter` 不可信；超界一律降级为 `matched_chapter=None + blocked=True`
  - `block_threshold` 在硬阻断 checker 里其实是**形参装饰**——只要 `score < 1.0` 就阻断，所以接收 `block_threshold` 但不用比较；标 `# noqa: ARG001` 保留 API 兼容（后续 thresholds_loader 注入需要）
  - report 同时暴露 `regex_match: bool` + `llm_match: bool|None` 让 evidence_chain 一眼看出走的哪条路径：`regex_match=True / llm_match=None` = regex 直通；`regex_match=False / llm_match=True` = LLM 救回；`regex_match=False / llm_match=False` = 双层都 miss
  - PRD 用例数严格按 `输出 5 passed` 验收，不要在测试里加额外 LLM 失败测试（虽然代码有 fallback 路径）；US-014 e2e 会再覆盖
  - `bool(matched_raw) if isinstance(matched_raw, (bool, int)) else False` 防御 LLM 返回字符串 `"true"`/`"false"`：JSON 严格 boolean 不会有这种问题，但 prompt 改写后偶尔会漂；isinstance 判一手保底
