- **AST 精确插入法**（US-002 `fix_utf8_encoding.py` 起）：修复 C1 用 `end_col_offset - 1` 定位闭合 `)`，插入 `, encoding="utf-8"`（有参）或 `encoding="utf-8"`（无参）；多行调用（`node.lineno != node.end_lineno`）一律保守跳过——避免破坏格式。幂等由复用 scanner 的 `_needs_encoding` 保证：修完后 `_has_encoding_kw` 返回 True。
- **C2 scanner context-aware 准则**（US-003 起）：纯字面量启发式扫描会产 110+ 误报（HTTP route / 中文显示标签 / dict 配置值 / sys.path bootstrap）。正确做法：必须验证字符串**真的被当作 filesystem 路径使用**——只在 ① fs call 参数（`Path()` / `os.path.exists/.join` / `shutil.copy` / `os.remove` 等已知白名单）② path-named target（`*_PATH`/`*_DIR`/`*_FILE`/`*_ROOT`/`*_FOLDER`）时报告。`os.path.join` 的非首参数永不报告（Python 在 Windows 上 normpath 归一化）。识别 CJK 段直接判定为显示标签。
- **`os.path.join` 跨平台安全保证**（US-003 起）：`os.path.join("/tmp", "a/b/c.txt")` 在 Windows 上 Python 会把 `a/b/c.txt` normalize 为 `a\b\c.txt`——所以 `os.path.join(..., '../ink-writer/scripts')` 这类 sys.path bootstrap 是 idiomatic 安全的，**不需要也不应该被 C2 报警**。`Path('a') / 'b/c'` 同理。
- **Path-named target 命名约定**（US-003 起）：跨平台代码里凡是用 `_PATH` / `_DIR` / `_FILE` / `_ROOT` / `_FOLDER` 后缀命名 filesystem 路径变量，C2 scanner 会自动 audit；新代码请遵循此约定，让审计能覆盖到。
- **subprocess 文本模式硬约束**（US-004 起）：所有 `subprocess.run/Popen/check_output/check_call/call`，只要带了 `text=True` / `universal_newlines=True` / `encoding=`，就**必须**显式 `encoding="utf-8"`。Windows 默认 cp936 解码 git/python 子进程的中文 stdout/stderr 必乱码。`tests/core/test_subprocess_cross_platform.py:test_repo_has_no_text_mode_subprocess_without_encoding` 是仓库级红线测试，引入新违规会直接挂 CI。
- **subprocess 禁用 shell=True**（US-004 起）：仓库当前 0 处真实使用；`test_repo_has_no_subprocess_shell_true` 守护这一约定。带空格路径或中文参数走 args 列表，`shell=True` 会导致 Windows 引号/拆词地狱。
- **AST last-arg-position 插入法**（US-004 `fix_subprocess_encoding.py` 起）：subprocess 调用大多是多行，不能像 fix_utf8_encoding 那样保守跳过——需要在"最后一个 positional arg / keyword value 之后"插入 `, encoding="utf-8"`，绕过关闭括号位置不确定问题。`max((arg.end_lineno, arg.end_col_offset) for arg in call.args + [kw.value for kw in call.keywords])` 拿到的位置一定在 `)` 之前，trailing comma 风格也兼容（变成 `value, encoding="utf-8",\n)`，valid Python）。这种插入法适用所有"在 call 末尾追加 kwarg"的 fixer。
- **asyncio Proactor 策略统一走 helper**（US-005 起）：所有生产入口调用 `runtime_compat.set_windows_proactor_policy()`（Mac 返回 False 无副作用，Windows 首次 True 并设 `_PROACTOR_POLICY_SET` 缓存，后续 True 幂等）。手写 `if sys.platform == "win32": asyncio.set_event_loop_policy(WindowsProactorEventLoopPolicy())` 等价块**不再允许**——scanner 按字面量检测 `set_windows_proactor_policy`，手写等价块会漏检。`tests/core/test_asyncio_proactor_policy.py:test_repo_non_test_asyncio_entries_declare_proactor_policy` 是仓库级红线测试，新入口缺字面量会直接挂 CI。
- **tests/conftest.py 是 asyncio 策略的单一真源**（US-005 起）：`tests/conftest.py` 在 module 加载时调 `set_windows_proactor_policy()`，pytest 在 collect 阶段先加载 conftest，下游所有 test 文件的 `asyncio.run(...)` 自动继承策略——无需每个 test 文件重复 boilerplate。`scan_c4_asyncio` 走 `_ancestor_conftest_covers_proactor` 父目录链寻找 conftest 的 token 即判定已覆盖。若 token 在 conftest 里被别名 import（`as _x`），静态 contains 检测仍通过，但若想升级到严格 AST call 检测（参考 `tests/core/test_asyncio_proactor_policy.py:test_tests_conftest_declares_proactor_policy`），请**不要**在 conftest 里用 alias——保持 `from runtime_compat import set_windows_proactor_policy` 的命名可识别。
- **safe_symlink 是 symlink 的单一入口**（US-006 起）：所有生产/一般 test 代码调用 `runtime_compat.safe_symlink(src, dst)`（Mac/Linux 走真实 `os.symlink` 保持字节级一致；Windows 无特权自动 `shutil.copyfile`/`copytree` 降级 + WARNING 日志）。**两个例外**：`_has_symlink_privilege` 的 probe 实现（本身就是探测 os.symlink 是否可用）+ `safe_symlink` 自己的实现——scanner 走 `_C5_PRIMITIVE_FUNCS` 跳过这两个函数体。**symlink 是 SUT 的场景**（如 test_path_guard 测 escape 拦截）用 `# noqa: c5` 或 `# c5-ok` pragma 显式标注，说明为何不用 helper——替换成 copy 会让 symlink 相关断言失效。`tests/core/test_safe_symlink.py:test_repo_has_no_bare_symlink_calls_outside_primitives` 是仓库红线，新违规直接挂 CI。
- **AST scanner 天然免疫字符串字面量误报**（US-006 起）：C5 从 line-regex 重写为 AST-based 后，`textwrap.dedent("""os.symlink(...)...""")` 这类 fixture 源码不再被扫到（ast.Constant 节点不参与 Call 匹配）；docstring 里的 ```os.symlink``` 文本同理跳过。**此改造模板可复用**——C8（Python launcher 硬编码）若将来发现 docstring 误报，同样可以考虑 AST 重写。line-regex 的适用边界：纯脚本（.sh/.ps1/.md）或 Python AST 无法表达的格式（如 Makefile）。
- **`.ps1` 必须 UTF-8 BOM 且用 bytes 写入**（US-007 起）：PS 5.1 默认 ANSI 解码 `.ps1`，没有 `0xEF 0xBB 0xBF` 头部会让中文乱码。Write tool 不写 BOM，必须走 `path.write_bytes(b'\\xef\\xbb\\xbf' + content_bytes)`。`tests/scripts/test_script_entries_parity.py:test_every_ps1_sibling_has_utf8_bom` 是仓库红线，新 `.sh` 对等 .ps1 缺 BOM 会直接挂 CI。验证：`xxd <file> | head -1` 头 3 字节 `efbbbf`。
- **`tests/<name>/` 不能与顶层包同名**（US-007 起）：`tests/scripts/__init__.py` 存在时 pytest collection 会让 `tests/scripts` 成为 `scripts` 命名空间包的副本，覆盖顶层 `scripts/` 并让 `from scripts.migration import ...` 炸 `ModuleNotFoundError`。解决：`tests/scripts/` **不放** `__init__.py`（namespace package 形式）。未来若想建 `tests/ink_writer/` / `tests/benchmark/`（已存在）等同名目录必须遵循此约定。其他无同名冲突的目录（如 `tests/core/` vs 无 `core/` 顶层包）不受影响。
- **嵌套 git 仓库自动排除**（US-007 起）：`scripts/audit_cross_platform.py:_is_nested_git_repo_dir` + `tests/scripts/test_script_entries_parity.py:_is_nested_git_repo` 沿父目录链检测含 `.git/` 子目录的嵌套 clone（root 自身 .git 不计入），让审计跳过。**触发场景**：顶级 `/ralph/` 是 snarktank/ralph 外部参考 clone（`.gitignore:63 /ralph/`），不在本仓库跨平台承诺范围——不加此判定 audit 会报 `ralph/ralph.sh` 缺 .ps1 / `ralph/skills/ralph/SKILL.md` 缺 windows-ps1-sibling 等假 finding。未来再引入外部 clone 无需额外配置，只要它有自己的 `.git/` 就自动跳过。
- **.cmd 双击包装骨架**（US-007 起）：所有 `.sh` 的 `.cmd` 对等统一 4 行模板（与 env-setup.cmd / ink-auto.cmd 完全一致），新增 .sh 直接复用：
  ```
  @echo off
  REM <name>.cmd — Windows 双击/CMD 入口，转发到 PowerShell 版本
  powershell -NoProfile -ExecutionPolicy Bypass -File "%~dp0<name>.ps1" %*
  exit /b %ERRORLEVEL%
  ```
  `%~dp0` 是"本脚本所在目录含末尾斜杠"的批处理内置，比任何硬编码路径都稳。`tests/scripts/test_script_entries_parity.py:test_every_cmd_sibling_forwards_to_ps1` 做宽松校验（含 `powershell` + 对应 .ps1 文件名即可），不强绑参数细节以留未来定制空间。
- **SKILL.md 的 stem 级 parity**（US-008 起）：每个被引用的 `foo.sh` 必须在同一 SKILL.md 内有同 stem `foo.ps1` 引用——只检查"引用了任一 .ps1"还不够，要防"标记对齐但 .sh 和 .ps1 不是一对"的漂移。`tests/skills/test_skill_md_windows_parity.py:test_every_sh_reference_has_matching_ps1_stem_in_same_skill_md` 是仓库级红线，新增 SKILL.md 加 `.sh` 块时必须同文件补同名 `.ps1` 引用（例：引入 `run.sh` 必须也出现 `run.ps1` 字面量，不能只靠 `env-setup.ps1` 顶替）。
- **tests/skills/ 与 ink-writer/skills/ 的命名空间重名**（US-008 起）：pythonpath 含 `ink-writer`，而 `ink-writer/skills/` 是命名空间包；**不能**在 `tests/skills/` 创建 `__init__.py`——否则会覆盖前者。与 US-007 `tests/scripts` vs `scripts/` 的陷阱同源。Codebase Pattern "tests/<name>/ 不能与顶层包同名" 的完整适用范围现在应理解为"不能与任意 pythonpath 根下的包同名"。
- **Python launcher 统一走 bash/PS detector**（US-009 起）：`.sh` 内统一用 inline `find_python_launcher_bash` 函数 + `$PY_LAUNCHER` / `$PYTHON_LAUNCHER` 变量（`case OSTYPE` — `msys*/cygwin*/win32*` 探测 py -3 → python3 → python；`*)` 分支恒定 `python3` 保证 Mac/Linux 字节级一致）。`.ps1` 内用 `Find-PythonLauncher` 函数 + `$env:PYTHON_LAUNCHER`。**禁止**在这些 detector 函数体之外裸调 `python3` / `py -3`——`tests/core/test_python_launcher.py:TestRepoRedLineC8` 仓库红线直接跑 scanner 守护，新违规立即挂 CI。新增 `.sh`/`.ps1` 加 python 调用时必须先取到 launcher 变量再调。
- **`# c8-ok` pragma 语义**（US-009 起）：detector primitive 的字面量（candidates 数组、返回字符串、变量赋值）用行尾 `# c8-ok: detector primitive` 或 `# noqa: c8` 标注，scanner 跳过。bash 和 PS 都以 `#` 起注释，`_C8_NOQA_RE` 单正则跨语言工作。与 US-006 的 `# noqa: c5` 同构——"基础 primitive 需要字面量" 类免检机制的统一处理。
- **regex 抠出 bash 函数体注入子 shell 测语义**（US-009 起）：`_BASH_FUNC_RE = r'find_python_launcher_bash\s*\(\s*\)\s*\{.*?\n\}\n'`（DOTALL）从实文件抠出 detector 函数源码，通过 `bash -c "$SRC; find_python_launcher_bash; echo \$VAR"` 注入子 shell，模拟 `OSTYPE=darwin23/msys` 不同平台断言输出变量值。这比把函数源硬编码进 test 更抗漂移（源改测自动跟上），也比 `subprocess.run([./env-setup.sh])` 稳（规避 env-setup 的 side effects 如路径检测/export）。副作用风险：若函数大括号格式漂移（如末 `}` 不换行）需要更新 regex。可复用于其他 shell primitive 测试。
- **OSTYPE 判别 Windows git-bash / Mac / Linux**（US-009 起）：bash 内置 `$OSTYPE`，Mac=`darwin*`、Linux=`linux-gnu*`、Windows git-bash=`msys`、Cygwin=`cygwin`、MinGW=`win32`。用 `case "${OSTYPE:-}" in msys*|cygwin*|win32*) ... ;; *) ... ;; esac` 做跨平台分支比 fork `uname` 子进程更快、比 `$OSTYPE` 全值相等匹配更稳（避免遗漏版本后缀如 `darwin23`）。
