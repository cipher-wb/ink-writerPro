- **meta 字段是 chapter-level 信号归属**（US-009 起）：`pack["meta"]["scene_mode"]` 与 `pack["meta"]["chapter"]` / `pack["meta"]["injection_policy"]` 同级。后续再加 chapter-level 派生信号（例如 pov_mode / pacing_mode）一律挂 meta 而非 core/scene/global——保持 pack 消费者（writer/checker/polish）只需访问 `pack["meta"][...]` 即可拿到全部激活判定所需信号，不至于污染内容层（core/scene/global）。同时 pack 集成的最小 diff 模式 = 1 行 import + 1 行计算 + 1 个 key。
- **scene_mode 优先级单源**（US-009 起）：`ink_writer/core/context/scene_classifier.py:resolve_scene_mode` 是 7 值 scene_mode 的 canonical 解析函数。新增 scene 感知特性时须以该函数输出为唯一入参，不允许 checker/writer/polish 各自实现"基于章节号或关键词推断"的独立判定——违反会直接破坏 US-005/US-007 已建立的激活一致性。未来若真的需要 LLM judge 进阶推断，也应该以该函数为 fallback，绝不替代。
- **checker 行为豁免走 arbitration 而非改 spec 逻辑**（US-010 起）：当需要让既有 checker 在某场景软豁免特定 rule code 时，优先级策略是：(1) checker.md 只加一段 `## 直白模式阈值放宽 (v22 US-010)` 文字说明；(2) checker 自身实现保持不变（继续跑完整维度输出完整 issues）；(3) 豁免在 `arbitration.collect_issues_from_review_metrics` 阶段用白名单过滤完成。这样 (a) LLM checker 零 prompt drift 风险，(b) 回滚只需恢复 2 行 import，(c) 白名单集中在 `prose/directness_threshold_gates.py` 单源可 grep。适用于任何"场景感知但不改 checker 本体"的需求。
- **corpus-grounded 验收方法论**（US-011 起）：端到端验证当无法跑 live LLM 写章时的通用替代方案——(1) 在 `benchmark/reference_corpus` 里挑两组书作为目标文风 / 反面文风代理（本轮用 baseline 报告现成的"最直白 Top-5 vs 最华丽 Top-5"），跑量化 metric 证明新机制期望达成的目标是 achievable 且与反面区分显著；(2) 对机制本体（如 `simplify_text`）用合成 fixture 跑确定性 unit 测，锚定"机制自己能驱动文本朝目标移动" 的证据；(3) PRD 里凡是涉及 human-in-loop 的指标（盲测 / LLM judge）延至发版后 live run，记录方法论到 report 但标记 `passed=None` 非阻断。产物：`reports/prose-directness-verification.md` + `.json` + `scripts/verify_prose_directness.py` + `tests/scripts/test_verify_prose_directness.py`。关键陷阱：PRD 里"±10%"等紧约束可能成文在分布统计之前——实测发现不合理时按已落地 checker 的实际 green band 调整（M-4 从 PRD 原文 ±10% 放宽到 golden_three IQR [P25, P75]，严格对齐 mid_is_better 绿区判定），并在脚本 docstring 和报告里显式解释理由。
- **simplify_text fixture 工程化**（US-011 起）：设计触发多规则 + 满足 70% 字数下限的合成 fixture 需要平衡——(1) 三段式布局（心理 + 纯环境 + 对话混合），用 `\n\n` 分段匹配 `_compress_empty_paragraphs` 的 split 逻辑；(2) 空境段无人称代词（他/她/我/你/咱/它/们）也无对话引号才会激活压缩；(3) 抽象形容词占字数 20-25% 时能驱动 S1 删除但不触发 70% 回滚；(4) 有 `，` 的长句 >30 字才能触发 S2 拆分。经验值：fixture ~220 chars / 14 blacklist hits / 3 段可稳定产出 ~27% 缩短 + hits 14→0 + 3 规则命中。

  - **ruff I001 对"pytest + 同包 import 裸空行"敏感**：`import pytest\n\nfrom ink_writer... import ...` 里空行导致 I001；修复就是去空行 `import pytest\nfrom ink_writer... import ...`。ruff --fix 一键解决，但新增测试文件都该跑一次 `ruff check --fix` 再提交。

- **Learnings for future iterations:**
  - **corpus-grounded 验收是发版门禁的通用捷径**：需要 live LLM 的指标（主观盲测、新章实跑）可延至发版后 real-user 采集；能用 baseline 现成 Top-5 反差做"静态定向实验"的，一定比等 live run 快 10 倍且可重跑。后续 PRD 若有"新 vs 旧对比"需求默认套这个模式。
  - **PRD 里"±10%"这类硬约束要留余地**：成文时往往在分布统计之前，真实 IQR 可能更宽。验证脚本应按已落地机制的实际绿区判定、同时在 docstring 和报告里显式记录与 PRD 原文的偏差理由；不要闷头调 PRD 也不要硬刷数据
  - **simplify_text 合成 fixture 工程化技巧**：(a) 三段式 + `\n\n` 分段适配 empty paragraph 压缩 (b) 空境段无人称代词+无对话引号 (c) 抽象形容词 20-25% 保不触发 70% 回滚 (d) ≥30 字长句带 `，`,才能被 S2 拆。实测 ~220 chars / 14 hits / 3 段 = ~27% 缩短的 sweet spot
