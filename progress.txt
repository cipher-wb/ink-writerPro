## Codebase Patterns
- Schema 修改严格保持 `additionalProperties: false`：任何新顶层字段必须在 properties 显式声明，否则现存数据立即拒收。
- case_id pattern 容纳多前缀的方式：在主 alternation 段加分支（如 `LR(-[0-9]{4})?`），保留 `(M[0-9]+-)?` 与末尾 `[0-9]{4}$` 不动；每加一个前缀都要回归 `tests/case_library/test_schema_backward_compat.py::test_all_existing_cases_still_validate` 全 410 份。
- pytest.ini `testpaths` 是单行长串 — 新增目录追加在末尾即可，不要重排已有目录避免 diff 噪音；行后注释保持原样。
- ruff isort 要求：`from __future__ import annotations` 后空一行，再 stdlib，再空一行，再三方包；conftest/test 文件别忽略 `--fix` 自动修复。
- 测试用 `Draft202012Validator(load_schema(...))` fixture 模式：从 `tests/live_review/conftest.py` 取 `load_schema` fixture（按 schema 文件名加载），再实例化 validator；schema self-validity 用 `Draft202012Validator.check_schema(schema)`。
- Python 3.14 `datetime.utcnow()` 已 deprecated；新代码统一 `from datetime import UTC, datetime` + `datetime.now(UTC)`。ruff UP017 会提示。
- ID 分配复用 `case_library._id_alloc.allocate_case_id(cases_dir, prefix=...)`：传不同 prefix 自动隔离 counter file（`.id_alloc_<sanitized_prefix>.cnt`）+ glob 兜底防外部手工添加 yaml 让 counter 落后；并发安全（独占文件锁，4 worker spawn 0001..0004 无空洞已验证）。新 case 类型不需要再写 ID 分配器，写 `live_review.case_id.allocate_live_review_id` 这种薄封装即可。
- live-review 配置 dataclass 模式：`LiveReviewConfig` 是 frozen dataclass + 嵌套 `BatchConfig`/`InjectConfig`；`enabled=false` 在 `load_config` 内强制 `inject_into.{init,review}` 全 false（`& 与` 操作）— 这是关闭整个模块的 master switch，下游 inject point 务必查 `cfg.inject_into.<stage>` 而不是直接看 yaml。
- 资源路径必须锚定 `_REPO_ROOT = Path(__file__).resolve().parents[N]`：plan/示例里出现的 `Path("scripts/...")` / `Path("schemas/...")` 是相对 CWD 的脆弱写法，被 pytest fixture 调用时会随启动目录漂移。`ink_writer/live_review/*.py` 在 `parents[2]` 即 repo root；这条规矩适用所有 module-level 常量。

**Learnings for future iterations:**
- **Plan 中 case_id pattern 字面写错**：`^CASE-([0-9]{4}|LEARN|PROMOTE|LR)-(M[0-9]+-)?[0-9]{4}$` 只能匹配 `CASE-LR-NNNN`，匹配不了 `CASE-LR-2026-NNNN`（Plan Discovered Constraints 自己也说要兼容两种形式但提供的 regex 错了）。我改用 `^CASE-([0-9]{4}|LEARN|PROMOTE|LR(-[0-9]{4})?)-(M[0-9]+-)?[0-9]{4}$`，与 spec §3.2 / US-002 prefix='CASE-LR-2026-' / genre_acceptance.case_ids pattern 全部一致。后续 US 看到 case_id 时无需再 patch pattern。
- **case_schema.json 顶层 schema_version**：plan 提示不能加 `$schema_version` 字段（与 JSON Schema 自有字段冲突），但其实 case_schema.json 顶层 `additionalProperties` 没有显式 false（仅在 properties 内的对象限制），加自定义字段会被 jsonschema 当作元数据忽略。我没加任何顶层版本字段，逻辑版本号靠 commit message 描述（"schema_version 1.0→1.1"）。
- **live_review_meta.required 9 字段而非 10**：plan diff 中 `required` 列出 9 个字段（`source_bvid`/`source_line_range`/`score_raw`/`score_signal`/`verdict`/`title_guess`/`genre_guess`/`overall_comment`/`comments`），`score` 不在 required 中（因为 score 可为 null 时 LLM 可能会省略，但 score_raw 必填承载原始引语）。PRD AC 笔误说"9 字段"实际是指 required 9 个，properties 共 10 个。
- **ruff 未在 CI 上跑某些子目录**：跑 `ruff check schemas/` 不会报错（json 文件 ruff 不管），但 tests/ 下会触发 I001（import 排序）。新写测试文件时养成 `from __future__ import annotations` 后空行 → stdlib 空行 → 三方包的习惯，省一次 --fix。
- **pytest.ini testpaths 行尾注释**：原文件 testpaths 行结尾紧贴 `\n# 注：...` 注释，编辑时需用足够长的 old_string 锚定行末（我用 `tests/learn tests/_compat\n# 注：tests/editor_wisdom...`）才能精准追加 ` tests/live_review`。
