- **Learnings for future iterations:**
  - **LLM 返回 char_range 与 text 长度不一致是常态**：LLM 可能返 `text_len < char_range[1]-char_range[0]`（截断）或相反。非超长情况下**保留 LLM 原始 char_range**（信任 LLM 的源定位），只有超长切分时才按 sub.len 累加 sub_start。错误做法是一律从 text_len 重算，会让 happy path 测试失败（char_range[1] 被改成 len(text)）。
  - **`re.search(r"\{.*\}", text, re.DOTALL)` 是 LLM JSON 恢复的"双保险"**：许多 LLM（含 Haiku）偶尔会在 JSON 外包 ```json ... ``` 或前置解释文字。先 `json.loads(raw)` 快路径；失败再正则抓第一个 `{...}`；都失败才 None 进入重试。
  - **`_rechunk_oversize` 的句末边界阈值 `last_period > 100`**：保证切分点不过于靠近起始（否则第一块可能 < 200 违反 min_chunk_chars 语义）。如果最后一段（`cursor + last_period + 1 >= len(text)`）也要走 `cut=end` 避免把剩余拼成长尾。
  - **Haiku 入口不需要 `__main__`**：scene_segmenter.py 纯 library，无 CLI 入口，因此不需要 `enable_windows_utf8_stdio()` 调用；audit 红线只扫含 `if __name__ == "__main__":` 的 .py。

- 验收：pytest tests/case_library/test_cli_approve.py -v --no-cov → 3 passed；pytest tests/case_library + tests/audit + tests/core + tests/corpus_chunking + tests/qdrant --no-cov → 359 passed；pytest -q（全量 + cov）→ coverage 82.72% (≥ 70)；ruff check ink_writer/case_library/{approval,cli}.py tests/case_library/test_cli_approve.py → All checks passed；py_compile OK；`python3 -m ink_writer.case_library.cli approve --help` 输出正确
- **Learnings for future iterations:**
  - **jsonschema `Draft202012Validator.validate(data)` 与 `iter_errors(...)` 的取舍**：前者在第一个错误就 raise `ValidationError`（fail-fast、给 CLI 顶层 catch），后者返生成器可枚举全部错误。批量 YAML 场景"哪怕一个字段不合法都拒绝整批"→ validate() 即可，不需要也不该收集子错误（单 entry 缺 action 意味 YAML 结构不可信，全批拒绝比选择性应用安全）。
  - **"fail one entry, continue batch" 语义通过两层 try/except 实现**：顶层 `apply_batch_yaml` 里 `validate(data)` 让 schema 错误直接抛；per-entry 循环内 try 所有副作用（store.load/save/append_ingest_log），`CaseNotFoundError` 单独 catch 出更友好的 err 文案，其他异常走 `except Exception` 兜底 + `noqa: BLE001`。这样"100 entries 里 1 个 case_id 打错"不会让其他 99 挂掉。
  - **note 字段"有值时才写入 ingest_log event"**：若未传 note 则 event dict 无 "note" 键（而非 `"note": null`），jsonl 往 grep 里塞 null 会让下游统计误识 — 这种"只在有值时写入"模式比"null 表不存在"更友好。测试里 `assert "note" not in next(ev ...)` 固化这一契约。
  - **跨包 JSON schema 加载统一走 `_repo_root() / schemas / ...`**：approval.py 与 schema.py 都有各自的 `_repo_root()`（因路径层级相同都是 `parent.parent.parent`），相同 `lru_cache(maxsize=1)` 模式避免重复 IO。未来若要再加 schema，考虑抽到 `schemas/__init__.py` 做统一 loader（M5 再评估，M2 维持分散）。
  - **CLI `--batch` 用 `type=Path`**：argparse 直接把 str 转 Path，避免 `_cmd_approve` 内再转；`required=True` 让 argparse 自动报错 "error: the following arguments are required: --batch"，rc=2 不用手写 else 路径。
  - **approval.py 不需要 CLI 入口 → 不触发 audit 红线**：纯 library 无 `if __name__ == "__main__":`，因此 audit 的 `enable_windows_utf8_stdio` 检查不扫它（同 US-009 rules_to_cases.py）。audit 红线只扫含 main guard 的 .py。
  - **`store.load(case.case_id)` 返回新 Case 对象，修改后必须 `store.save(case)` 重写 YAML**：Case dataclass 是 value object，`case.status = new_status` 只改内存副本，磁盘文件不动直到 save。这是 CaseStore 设计的显式契约（load + mutate + save 三段式），与 ORM 的 autoflush 语义不同。
