**Learnings for future iterations:**
- **Plan 中 case_id pattern 字面写错**：`^CASE-([0-9]{4}|LEARN|PROMOTE|LR)-(M[0-9]+-)?[0-9]{4}$` 只能匹配 `CASE-LR-NNNN`，匹配不了 `CASE-LR-2026-NNNN`（Plan Discovered Constraints 自己也说要兼容两种形式但提供的 regex 错了）。我改用 `^CASE-([0-9]{4}|LEARN|PROMOTE|LR(-[0-9]{4})?)-(M[0-9]+-)?[0-9]{4}$`，与 spec §3.2 / US-002 prefix='CASE-LR-2026-' / genre_acceptance.case_ids pattern 全部一致。后续 US 看到 case_id 时无需再 patch pattern。
- **case_schema.json 顶层 schema_version**：plan 提示不能加 `$schema_version` 字段（与 JSON Schema 自有字段冲突），但其实 case_schema.json 顶层 `additionalProperties` 没有显式 false（仅在 properties 内的对象限制），加自定义字段会被 jsonschema 当作元数据忽略。我没加任何顶层版本字段，逻辑版本号靠 commit message 描述（"schema_version 1.0→1.1"）。
- **live_review_meta.required 9 字段而非 10**：plan diff 中 `required` 列出 9 个字段（`source_bvid`/`source_line_range`/`score_raw`/`score_signal`/`verdict`/`title_guess`/`genre_guess`/`overall_comment`/`comments`），`score` 不在 required 中（因为 score 可为 null 时 LLM 可能会省略，但 score_raw 必填承载原始引语）。PRD AC 笔误说"9 字段"实际是指 required 9 个，properties 共 10 个。
- **ruff 未在 CI 上跑某些子目录**：跑 `ruff check schemas/` 不会报错（json 文件 ruff 不管），但 tests/ 下会触发 I001（import 排序）。新写测试文件时养成 `from __future__ import annotations` 后空行 → stdlib 空行 → 三方包的习惯，省一次 --fix。
- **pytest.ini testpaths 行尾注释**：原文件 testpaths 行结尾紧贴 `\n# 注：...` 注释，编辑时需用足够长的 old_string 锚定行末（我用 `tests/learn tests/_compat\n# 注：tests/editor_wisdom...`）才能精准追加 ` tests/live_review`。

- **`_failed.jsonl` 必须无视 `--skip-failed` 都写盘**：即使非 `--skip-failed` 路径（break 退出 + return 1），failures 也要追加到 `_failed.jsonl`，方便用户事后查 traceback。我把 `_append_failed` 调用放在 `if failures:` 块（不在 `if args.skip_failed:` 块内）— 这是 b2 测试通过的关键之一（虽然 b2 只断言 returncode=1，但 b1 必须看到 _failed.jsonl 的出现）。

- **`except Exception` 的窄化**：BLE001 警告原本希望 except 子类化。但 batch 跑批的全 fault-tolerance 设计本意就是吃掉所有异常（mock IO / read_text 编码 / json parse / schema 校验），所以加 `# noqa: BLE001` 是合理的（Plan 字面 `except Exception as e` 的写法认可此宽口径）。如果未来缩窄到 `(ExtractionError, FileNotFoundError, json.JSONDecodeError)`，注意 schema 校验失败现在被 ExtractionError 包装，但 OS 层错误（如 `PermissionError`）会逃出 — 工程取舍。
- **complete_mock_dir 复合 fixture 模式**：有些场景需要 5 mock 全齐（a/c/d），有些需要故意缺 1 mock（b）。在 conftest/test 内部定义 `complete_mock_dir` fixture：拷贝 mock_batch 4 mock + tmp_path 合成 1 minimal record 即可。minimal record 用 `score=None / score_signal=unknown / verdict=unknown / comments=[]` 通过 schema 校验。后续类似"全集 vs 缺漏"的对比测试可复用此 fixture 模板。

- **polish 触发但不重试**：editor_wisdom.review_gate 是"polish + 重检最多 3 次"的完整循环；live_review.review_injection 只触发一次 polish，重试由调用方控制。理由：(1) 黄金三章 0.75 vs 0.65 vs 阈值差距小，单次 polish 已能修大部分违规；(2) 与 editor-wisdom OR 并列后总体重试次数已经够；(3) 测试断言简单（polish_triggered True/False + polish_calls 长度）。如果未来需要 retry 循环，可在 review_injection 内部加 `max_retries` 参数（参考 editor_wisdom.review_gate）。
- **agent spec 5 段标准结构**：Purpose / Input / Retrieval / Scoring / Output —— 测试 `test_agent_spec_exists_and_has_required_sections` 用 5 个 `assert section in text` grep。这是与 33 个现有 checker spec 对齐的最小集，后续新加 checker spec 沿用此模板。`{{PROMPT_TEMPLATE:checker-input-rules.md}}` 必须放在文件开头第一段（紧跟 frontmatter + agent 描述段）。
- **score 在 mock 与计算路径上的不一致**：PRD agent spec §1 说"综合 = (1 - violation_density) × verdict_pass_rate_of_top5"，但实现里 `run_live_review_checker` 直接用 LLM 返回的 score 不再二次计算。这是因为：(1) LLM 已能按公式输出 score，二次计算可能与 LLM 自评分歧造成 evidence_chain 自相矛盾；(2) 公式作为 LLM prompt 的指导显式写在 prompt 中，让 LLM 内部按规则评分；(3) mock 模式下 score 来自 fixture，二次计算反而破坏 fixture 设计意图。后续如果发现 LLM 评分漂移，可加 `--strict-formula` flag 在 Python 端二次校验，但目前不必。
- **配置 master switch 在两层模块都查**：`run_live_review_checker` 与 `check_review` 都查 `cfg.enabled / cfg.inject_into.review`。看似冗余，但用户可能直接调 checker（绕过 review_injection），单独的 checker 也必须支持关闭。两层独立查同一开关与 init_injection.check_genre 模式一致，是 live-review 模块的统一约定。后续 US-013 smoke_test 也会走 checker 直接路径，需要这层防护。
- **SKILL.md Python 块字符串测试不细究 markdown 结构**：用 `assert "Step 3.6" / "live_review" / "inject_into" / "OR | 并列" in text` 5 层 grep 防 SKILL.md 被回滚；不 assert 完整代码块的具体行（否则微调即破坏 test）。这套"消极保险"测试与 US-011 的 SKILL.md 测试同款，sustainable。

**Learnings for future iterations:**
- **subprocess 测试与 module-scope fixture 共享 vector index 的模式**：subprocess 进程不能直接消费 pytest fixture，但脚本接受 `--index-dir <pre-built-path>` 参数即可让 fixture 中预构建的索引被复用。`tmp_path_factory.mktemp(scope="module")` 让 4 个 subprocess test 只付一次 30s 构建成本（约 35-45s 总），相比每 test 一次构建（4×30s=120s）节省 65%+。这是所有"subprocess 跑脚本但脚本依赖重型外部资源"的通用优化模式。后续 US-014 / 其他 e2e smoke 沿用此模板（fixture 预构建 + 脚本 `--<resource>-dir` 接住）。
- **mock 模式 review checker 完全不读 ANTHROPIC_API_KEY**：`run_live_review_checker(..., mock_response=mock)` 在 `_validate_result_shape` 之后立刻 return，从不进入 retrieve_similar_cases 也不调 anthropic SDK。这意味着 smoke_test.py 默认模式（无 `--with-api`）即使在没有 API key 的 CI 上也能通过——这是 plan §M-9 "ralph 机器可验，无真实 API" 约束的关键支点。后续任何 e2e smoke 默认走 mock 路径必须严格遵守"mock_response 直返不进 retrieval"语义。
- **report markdown 含 'PASS' 关键字的稳定写法**：用 dataclass `StepResult(status='PASS')` + 渲染表格 + `All checks PASS` 摘要行。即使任一 step FAIL，仍保留通过 step 的 `| step_X | PASS | ... |` 行 + 改尾行为 `N checks FAILED`。AC "内容含 'PASS' 关键字" 在 ramen-soup 场景仍可能命中（PASS substring in 'FAILED' 倒也不出现，但 `| PASS |` 表格单元格永远存在），所以更严格的"All checks PASS"摘要行作为权威信号。后续报告类 AC 写法沿用此双层（表格单元格 + 显式摘要句）。
- **`--index-dir` 默认指向 `data/live-review/vector_index/` 会污染 repo**：用户首次跑 smoke_test 不带参数 → 触发首次构建到 `data/live-review/vector_index/`。这与 build_vector_index.py 的默认输出一致（脚本设计就是让 smoke_test 同时初始化用户的实用索引），但测试场景必须显式 `--index-dir <tmp_path>` 避免污染。这是 spec §M-1 "用户首次跑 smoke 即建初始索引"语义的实现，不是 bug。
- **dataclass 字段不要漏写默认值**：`StepResult(detail: str = "")` 必须显式默认空串否则部分 step 不传 detail 会 TypeError。Plan 字面没强调但是 mypy/ruff 不查这种运行时遗漏，必须 `pytest -v` 跑过才暴露。后续 dataclass 定义先写默认值再写代码，避免"等 RED 再加默认值"的重复迭代。
- **`# noqa: BLE001` 用法**：smoke_test 三个 step 函数都用 `except Exception as e: ... return StepResult("FAIL", ...)`——故意吞所有异常存到 detail 字段，给报告诊断用。ruff BLE001 默认禁止 broad except，加 `# noqa: BLE001 — surface ... in report` 行级注释 + 注明意图即可。后续所有"先写报告，最后由 main 决定退出码"的脚本沿用此 fail-soft 模式。
- **`from datetime import UTC, datetime` 是 Py 3.11+ 的标准写法**：`datetime.now(tz=UTC)` 替代 `datetime.utcnow()`（后者在 3.14 deprecated）。smoke_test.py 报告时间戳沿用此写法与 editor-wisdom/smoke_test.py 一致。
