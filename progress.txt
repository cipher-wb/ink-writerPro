## Codebase Patterns
(承接自前轮 ralph runs，保留通用、适用本 PRD 的条目。逐轮新增追加到下方。)

- **UTF-8 编码硬约束**：所有 `open()` / `Path.read_text()` / `Path.write_text()` 必须带 `encoding="utf-8"`（见 CLAUDE.md Windows 兼容守则第 1 条）。中文项目在 Windows 上不带参数会炸 UnicodeDecodeError。
- **零回归守卫 baseline**：v18.0.0 发版后 `pytest --no-cov` 全量 baseline = 2984 passed / 19 skipped / 0 failed。本轮每个 US 完成前必须跑全量无新增失败。
- **pytest.ini testpaths 同步**：新增的 `tests/核心模块/子目录/` 必须追加到 `pytest.ini:testpaths`，否则 `python3 -m pytest` 默认收集会漏掉。
- **ProjectConfig 读取约定**：章节正文文件命名/后缀由 `ProjectConfig` 决定（`benchmark/reference_corpus/*/chapters` 使用 `ch###.txt`，实际项目可能不同）；新代码不硬编码命名，统一从 config 读取。
- **ralph/ 是 gitignored**：`ralph/prd.json` 与 `ralph/progress.txt` 的状态翻转与 append 都不入 commit；正式归档副本在 `archive/YYYY-MM-DD-feature/` 下，受版本控制。
- **kwarg-only 向后兼容**：给既有 public 函数新增参数时放到 `*` 之后（kwarg-only），避免破坏生产/测试调用方的位置参数假设。`ink_writer/editor_wisdom/writer_injection.py:build_writer_constraints` 是典型示例。
- **章节文件查找统一入口**：定位章节正文文件必须用 `chapter_paths.find_chapter_file(project_root, chapter_num)`（lazy import，兼容两种导入路径 `chapter_paths` / `scripts.chapter_paths`），它已覆盖三种命名约定（`第NNNN章.md` 平坦布局、`第N卷/第NNN章-标题.md` 卷布局、rglob fallback）。新代码不要 hardcode `f"ch{n:04d}.txt"` 之类。
- **summary 文件命名约定**：`self.config.ink_dir / "summaries" / f"ch{chapter:04d}.md"`（见 `context_manager.py:_load_summary_text`），与正文章节文件命名完全不同；测试里 `_write_summary` 需要走 `ink_dir/summaries`，不要放到 `chapters_dir`。
- **context_contract_version 锁死 v3**：`context_ranker.py:48` 硬编码 v3，多处测试断言 v3；本 PRD 新增字段一律**additive**，不要随手 bump v4 否则回归立刻炸。
- **"硬注入 = protected section" 模式**：US-006 落地的规范——想实现"某 section 永不字符级裁剪"，**不要**给它 `TEMPLATE_WEIGHTS` 里配高权重（weight > 0 必然触发 budget = max_chars * weight 的字符级裁剪）；正确做法是把 section 名加入 `context_protected_sections` config → `assemble_context` 对 protected section 直接 budget=None，`_compact_json_text(content, None)` 返回原 JSON 不裁剪。trim 循环里 `_trim_section` 第一行 short-circuit 跳过 protected。未来任何"硬注入"字段都复用这套机制，别自造车轮。
- **虚拟 top-level section（嵌套字段抬升）**：若要把 `pack.core.xxx` 这种嵌套字段在 assembled 输出里作为独立 section 呈现，在 `SECTION_ORDER` 里登记虚拟名（如 `recent_full_texts`）、在 assemble 循环里显式 `if section_name == "<virtual>": sections[...] = pack["core"].get("<virtual>", []); continue`，同时构造不含该字段的 `core_for_render` 副本走 weight 路径避免双重计算；pack 原始结构绝不修改（US-002 的外部消费者仍读原 schema）。本模式见 `context_manager.py:181-210`。

## Context Contract（本 PRD 专属）

- **核心数据契约**：context pack 新增 `recent_full_texts: List[{chapter, text, word_count, missing}]`，语义为 n-1/n-2/n-3 三章全文；`recent_summaries` 语义变更为 n-4~n-10（不再重叠全文范围）。

- **Learnings for future iterations:**
  - **protected 语义 vs weight 语义**：原 PRD AC 提"context_weights.py 权重重排"，直觉上是改 `TEMPLATE_WEIGHTS` 给 recent_full_texts 配高权重。但 weight 机制本质是 `budget = max_chars * weight`，任何 `weight > 0` 都会触发字符级裁剪，无法表达"永不裁剪"。所以正确的"权重重排"是**绕过 weight 机制**：用 `context_protected_sections` 这个 config 告诉 assemble_context 直接 budget=None。未来任何 section 要"硬注入"都沿用这个约定，不要再试图用 weight=1.0 或 weight=很大数值 间接实现。
  - **pack vs assembled 契约分离**：assemble_context 可以任意重写 `assembled.sections`（包括 lift、shrink、重写 text），但**绝不能修改原 pack**。本 US 在 lift full_texts 时特意 `{k: v for k, v in core_original.items() if k != "recent_full_texts"}` 做浅拷贝副本，确保 `pack.core.recent_full_texts` 不被破坏（US-002 schema 的外部消费者仍能读到）。测试 `test_core_section_no_longer_contains_full_texts` 断言了这一分离。
  - **SECTION_ORDER 里的"虚拟顶层 section"**：`recent_full_texts` 在 SECTION_ORDER 中登场，但在 pack 里并**不**存在同名顶层键（它嵌套在 `core` 下）。assemble_context 循环里显式 `if section_name == "recent_full_texts": sections[...] = recent_full_texts; continue` 来处理这种"虚拟 section"。未来若有类似"嵌套字段想独立展示"的需求，沿用这套模式即可，不要把 pack 结构改扁平。
  - **hard_cap 降级中 recent_summaries 的特殊处理**：recent_summaries 嵌在 `core.content` 里（与 chapter_outline/protagonist_snapshot 并列），直接 `_trim_section("recent_summaries")` 无从切入（没有同名顶层 section）。解法是在 hard_order 循环里给它特殊分支：读 `core_sec.content`（原生 dict）、构造 `shrunk = {..., "recent_summaries": []}`、用 `_compact_json_text(shrunk, core_sec.budget)` 重写 text。这样既保留了 core 的 text 结构，又精准去除了 summaries 数据。后续若有其他"嵌套字段单独降级"需求沿用这个模式。
  - **trim_stages_applied 日志字段的诊断价值**：返回 `meta.trim_stages_applied = ["soft:alerts", "soft:preferences", "hard:global"]` 这种有前缀的字符串，比单纯 `budget_trimmed=True` boolean 更能帮运维/调试判断到底降级了多少。未来其它类似的"多阶段裁剪"机制（比如未来可能出现的 RAG 结果裁剪）都应沿用 `<tier>:<target>` 的命名。
  - **默认 soft/hard 60k/180k 远高于现实 pack 大小**：3×2500 字全文 ≈ 7500 字 ≈ 5000 tokens；加上 summaries (~5k) + outline (~1k) + global (~2k) + 其他 ≈ 15k~20k tokens，远低于 60k soft。所以默认配置下，**绝大多数写作场景不会触发任何降级**——soft/hard 是"出现异常时的兜底"而非"日常工作机制"。未来如果看到大量 `meta.budget_trimmed=True` 报告，要先排查是不是 outline/scene 异常膨胀了，而非加激进降级。
