## Codebase Patterns
- 零容忍规则现在有 `kind` 字段：`regex`（默认，向后兼容历史规则）或 `density`（按 occurrences/千字 阈值阻断）；新规则在 `config/anti-detection.yaml` 加 `kind: density` + `density_threshold` 即可，loader 自动处理。
- 所有 ZT 规则都带 `whitelist_patterns: []` 字段（loader 强制存在，运行时尚未消费）；后续若要做"对话被打断豁免/引用片段豁免"在 `_rule_violated` 内补豁免逻辑即可，YAML schema 不用动。
- `check_zero_tolerance` 走 first-match 短路：YAML 顺序决定哪条规则先报；测试 density 规则的细分行为时用 `AntiDetectionConfig(zero_tolerance=[single_rule])` 隔离，比给定 fixture 同时违反多条规则更可控。
- 测试样本里的 `「」` (U+300C/D) 是中文方头括号，**不**在 `ZT_AI_QUOTES` 黑名单内（黑名单只针对 U+201C/D/2018/9 + «»）；这是合规的爆款引号风格。
- `prose-blacklist.yaml` 现在有 6 大类 + replacement_map（version 2）；新增三类（pretentious_verbs/nouns/adverbs）通过 `CATEGORIES` 元组扩展自动落入 `Blacklist.entries` —— 不需要改 `_parse_entry`/`match`，只要把新类目加到 `CATEGORIES` 就被加载。
- `Blacklist` 数据类增加了 `replacement_map: ReplacementMap` 字段（默认空）；equality 仍然成立因为 frozen dataclass 字段比较走 `__eq__`，旧测试 `Blacklist(version=0, entries=())` 与 `_empty()` 仍然相等。
- `ReplacementMap.lookup()` 取首项是 polish-agent simplification_pass 的"机械替换"约定；YAML 顺序决定首项 → 改顺序 = 改默认替换。`origins(replacement)` 反查可服务"语义聚类"（盯着 ← 凝视 + 凝望 + 凝眸）。
- 词表新加词时，**必须**先验证不与 abstract_adjectives/empty_phrases/pretentious_metaphors 重复（test_no_word_collisions_across_pretentious_categories 强制三个新域互斥；与老三域共存的检查靠 spot review）。
- 改 prose-blacklist.yaml 时 simplification_pass 测试 fixture **可能**被新词命中（test_redundant_paragraph_reduced_and_blacklist_cleared 里"苍茫"被吃进 pretentious_nouns 后就破了）；S1 规则只清 abstract_adjectives，新域要等 US-003 接 replacement_map 才会被自动剥除。
- `simplify_text` 流程现在是 S1 (abstract_adj 删除) → S1.5 (replacement_map 替换) → S2/S3/S4 (拆句/修辞/空段) → 70% floor。S1 与 S1.5 词典互斥（YAML 契约 + US-002 测试守护），不会双计或互相抵消。
- `apply_replacement_map(text, *, replacement_map=None, chapter_no=None, diff_dir=None)` 是独立可复用 API：source words 按字符长度倒序处理避免 1-char 前缀误吃 2-char 词；多候选取 `lookup(word)[0]`（YAML 列表首项）；source==replacement 时跳过；空跑或 diff_dir 缺失时不写文件。
- diff 文件命名 `chapter_{NNN:03d}.diff`（3 位补零）；同章节多次跑 polish 后写覆盖前写（避免无限增长）；payload = 头注释命中清单 + unified diff body。如果未来要保留多版本，加 timestamp 后缀但需同步更新 polish-agent.md 文档。
- `SimplificationReport` 新增 `replacement_hits: tuple[(src, rep, count), ...]` + `replacement_diff_path: str | None` 字段。回滚（rolled_back=True）时这两个字段仍保留中间状态供审计——回滚只抹掉 simplified_text 不抹掉元数据。

- **Learnings for future iterations**:
  - 测试 simplify_text 配合自定义 Blacklist 时，fixture 长度要够大避免触发 70% floor 回滚（删 4 字节后 simplified_char_count / original_char_count 必须 ≥ 0.70 才不回滚）；参考 `test_simplify_text_custom_blacklist_no_replacement_map` 用 24 字 fixture 删 4 字保留 83%。
  - `apply_replacement_map` 默认实参 `replacement_map=None` 时 lazy 加载 `load_blacklist().replacement_map`；这意味着函数可不依赖外部 fixture 直接调用，但测试隔离时仍应 `clear_cache()` 防 YAML mtime 缓存命中污染。
  - longest-first tie-break 用 `sorted(words, key=lambda w: (-len(w), 0))`：负号反转长度排序；副 key 为 0 是占位（未来按 frequency 排序时改这里）。
  - diff 写入用 `difflib.unified_diff` 而非自建格式：lineterm="" + splitlines(keepends=False) 保证跨平台行为一致；加 `# header` 行不会破坏 unified diff 解析（Diff 工具默认忽略以 # 开头的注释）。
  - S1.5 阶段的命中清单 `replacement_hits` 即使在 70% floor rollback 时也保留——这是审计需求（让 polish-agent 报告完整反映"想做了什么"，rollback 只影响最终 simplified_text 不影响元数据）。同样的契约也适用于 `replacement_diff_path`：diff 文件已落盘了不会因 rollback 删除。
  - `ReplacementMap` 的 `forward.keys()` 顺序在 Python 3.12+ 是插入顺序（`dict.__iter__` 保证），即 YAML 解析顺序；这是 polish-agent "YAML 顺序 = 优先级" 契约的实现层依赖。
  - 命中清单格式选择 `tuple[tuple[str, str, int], ...]` 而非 `list[dict]`：frozen + 易序列化 + 测试断言用 `("凝视", "盯着", 1) in result.hits` 直接成立。
