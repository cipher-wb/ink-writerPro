## Codebase Patterns
(承接自前轮 ralph runs，保留通用、适用本 PRD 的条目。逐轮新增追加到下方。)

- **UTF-8 编码硬约束**：所有 `open()` / `Path.read_text()` / `Path.write_text()` 必须带 `encoding="utf-8"`（见 CLAUDE.md Windows 兼容守则第 1 条）。中文项目在 Windows 上不带参数会炸 UnicodeDecodeError。
- **零回归守卫 baseline**：v18.0.0 发版后 `pytest --no-cov` 全量 baseline = 2984 passed / 19 skipped / 0 failed。本轮每个 US 完成前必须跑全量无新增失败。
- **pytest.ini testpaths 同步**：新增的 `tests/核心模块/子目录/` 必须追加到 `pytest.ini:testpaths`，否则 `python3 -m pytest` 默认收集会漏掉。
- **ProjectConfig 读取约定**：章节正文文件命名/后缀由 `ProjectConfig` 决定（`benchmark/reference_corpus/*/chapters` 使用 `ch###.txt`，实际项目可能不同）；新代码不硬编码命名，统一从 config 读取。
- **ralph/ 是 gitignored**：`ralph/prd.json` 与 `ralph/progress.txt` 的状态翻转与 append 都不入 commit；正式归档副本在 `archive/YYYY-MM-DD-feature/` 下，受版本控制。
- **kwarg-only 向后兼容**：给既有 public 函数新增参数时放到 `*` 之后（kwarg-only），避免破坏生产/测试调用方的位置参数假设。`ink_writer/editor_wisdom/writer_injection.py:build_writer_constraints` 是典型示例。
- **章节文件查找统一入口**：定位章节正文文件必须用 `chapter_paths.find_chapter_file(project_root, chapter_num)`（lazy import，兼容两种导入路径 `chapter_paths` / `scripts.chapter_paths`），它已覆盖三种命名约定（`第NNNN章.md` 平坦布局、`第N卷/第NNN章-标题.md` 卷布局、rglob fallback）。新代码不要 hardcode `f"ch{n:04d}.txt"` 之类。
- **summary 文件命名约定**：`self.config.ink_dir / "summaries" / f"ch{chapter:04d}.md"`（见 `context_manager.py:_load_summary_text`），与正文章节文件命名完全不同；测试里 `_write_summary` 需要走 `ink_dir/summaries`，不要放到 `chapters_dir`。
- **context_contract_version 锁死 v3**：`context_ranker.py:48` 硬编码 v3，多处测试断言 v3；本 PRD 新增字段一律**additive**，不要随手 bump v4 否则回归立刻炸。

- **Learnings for future iterations:**
  - **Issue schema 扩展与 drift_detector 的并存约定**：`ink_writer/propagation/drift_detector.py:45` 的 `_CHAPTER_KEYS = ("target_chapter", "ref_chapter", "source_chapter", "chapter")` 已原生支持顶层 `source_chapter` 字段读取，所以第五层新增的 `evidence.source_chapter` 不冲突但也不会被 drift detector 读。规范要求 **同时保留顶层 `ref_chapter` / `source_chapter`**（供 drift_detector 反向传播）+ **额外附加 `evidence.{source_chapter, excerpt}`**（供 fix / review 做精准片段跳转），两者是互补关系，不是替代关系。未来扩展任何 checker schema 时，都要先 grep drift_detector 的 `_CHAPTER_KEYS` / `_DRIFT_CHECKERS` / `_CROSS_CHAPTER_TYPES` 确认向后兼容。
  - **"降级兜底链"贯穿三 agent**：context-agent 的 `full_text_injection_failed:<reason>`（IO 异常） + `missing_full_text:chNNNN`（结构性缺文件）+ continuity-checker 的 `evidence_source=degraded:no_full_texts`（整体缺失）+ `n1_no_prior`（N=1 无前置）形成一致兜底链。消费者拿到 `evidence_source` 值就能判定是否信任 evidence；消费者看到 `missing_full_text_chapters` 就能判定第五层覆盖度。未来任何新 agent 接入 `recent_full_texts` 体系都应沿用这一兜底词汇表，不要另造新值。
  - **"additive 不 bump 版本"继续有效**：与 US-002 一致，本 US 只向 `checker-output-schema.md` 的 continuity metrics 添加字段、向 issues[] 添加 evidence 子字段，不 bump `context_contract_version` 也不引入新 schema 版本号。消费者侧做 `metrics.get("evidence_source", "legacy")` / `issue.get("evidence")` 即可兼容老缓存。
  - **"第五层而非替代四层"很关键**：AC 明确要求"不破坏 checker 已有的 review_metrics / character_evolution_ledger 校验路径（与新机制并存）"，规格里通过 "五层" 并排 + `related_issue_ids` 合并条款落实。如果把第五层写成"替代前四层"，会丢 active_threads 主线/支线的监控能力，对项目中段（50+ 章）影响巨大。未来类似 checker 升级记住：**新机制是加法，不是替换**。
  - **spec-only US 的回归验证硬约束（延续）**：纯 Markdown 改动无新增测试仍必须跑全量 pytest 证明零回归（feedback_no_regression 硬规则）；US-005 回归输出与 US-004 逐位一致（3008/23/0），证明 spec 改动未触发任何 pytest 动态加载/模板渲染/CI 的意外路径。未来 spec-only 批量改动也应采用"baseline 位对位比对"作为通过阈值。
