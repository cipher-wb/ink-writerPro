# v9.0 升级与迁移指南

> 从 v8.x 升级到 v9.0 Harness-First 架构的完整说明。

---

## 一句话总结

**不迁移也能用。** 所有旧命令照常工作，只是享受不到 v9.0 的新功能（读者评分趋势、增强报告）。

---

## v9.0 改了什么

### 命令面收口：13 → 4 主入口

你只需要记住 4 个命令：

| 命令 | 用途 |
|------|------|
| `/ink-init` | 创建项目 |
| `/ink-auto N` | **日常写作**（90% 的使用场景） |
| `/ink-resume` | 中断恢复 |
| `/ink-migrate` | 旧项目升级（新增） |

其他命令（`ink-write`、`ink-review`、`ink-plan` 等）全部保留，降级为高级/调试命令。

### 计算型闸门（Step 2C）

在每章的 LLM 审查（Step 3）之前，新增一道确定性规则检查：字数、文件命名、伏笔逾期、契约完整性等。硬失败直接退回重写，不浪费 checker 调用。

**对你的影响**：零。自动运行，脚本不存在时静默跳过。

### Reader Agent 升格

`reader-simulator` 从"只在重要章节跑"升级为"每章必跑"的核心裁判。

输出 7 维结构化评分（`reader_verdict`）：

| 维度 | 说明 |
|------|------|
| hook_strength | 开头抓取力 |
| curiosity_continuation | 中段好奇心维持 |
| emotional_reward | 情绪回报感 |
| protagonist_pull | 主角吸引力 |
| cliffhanger_drive | 章末追更驱动 |
| filler_risk | 注水风险（反向） |
| repetition_risk | 套路重复风险（反向） |

总分 ≥32 通过，25-31 自动增强，<25 退回重写。

**对你的影响**：每章多 1-2 分钟审查时间，换来"写出来的东西有人想看"的保证。

### ink-auto 增强输出

跑完 `/ink-auto N` 后，报告从"写了几章、审了几次"升级为：

```
质量概览：
├─ 平均审查分：82/100
├─ 读者评分趋势：↑ 稳步上升
├─ 自动修复：7 次
└─ 未修复残留：0 项

追读力信号：
├─ 活跃伏笔：12 条
└─ 逾期伏笔：1 条 ⚠️

推荐下一步：ink-auto 10
```

## 迁移方法

### 你需要迁移吗？

| 你的情况 | 建议 |
|---------|------|
| 还没开始写，准备新建项目 | **不需要迁移**。`/ink-init` 直接创建 v9.0 项目 |
| 写了一半，想继续用 | **建议迁移**。3 分钟搞定，享受全部新功能 |
| 快写完了，不想折腾 | **不需要迁移**。所有旧功能正常运行 |

### 迁移步骤（3 分钟）

#### 第一步：进入项目目录

```bash
cd /你的小说项目目录
```

确保目录下有 `.ink/state.json`。

#### 第二步：运行迁移

```bash
/ink-migrate
```

系统自动执行三个阶段：

**Phase 1：资产发现**（只读扫描，不改任何文件）
- 扫描章节文件、摘要、大纲、审查报告
- 检查 state.json 和 index.db 状态
- 输出资产清单

**Phase 2：Schema 迁移**
- 自动备份整个 `.ink/` 目录到 `.ink/backups/pre-migration-v9-{时间戳}/`
- state.json 从 v6 升级到 v7（新增 `harness_config` 字段）
- index.db 创建 2 张新表（`harness_evaluations` + `computational_gate_log`）

**Phase 3：迁移审计**
- 生成审计报告到 `.ink/migration/audit_report.md`
- 列出高/中/低置信度项
- 给出推荐的后续操作

#### 第三步：验证

```bash
/ink-auto 1    # 写 1 章测试
```

正常完成 = 迁移成功。

### 迁移完成后的变化

| 内容 | 变化 |
|------|------|
| 章节正文 | **不变**。一个字都不改 |
| 大纲文件 | **不变** |
| state.json | 新增 `harness_config` 和 `schema_version: 7` |
| index.db | 新增 2 张表 |
| 旧命令 | 全部可用 |
| 新功能 | reader_verdict 趋势分析、增强报告 |

### 回滚（万一出问题）

迁移前自动备份了，回滚很简单：

```bash
# state.json 回滚
cp .ink/state.json.bak.6 .ink/state.json

# index.db 回滚（使用迁移前备份）
# 备份位置在迁移输出中有打印
```

## 常见问题

**Q: 迁移会不会改我已经写好的正文？**
A: 不会。迁移只动 `.ink/` 目录下的数据库和配置文件，章节正文一个字都不碰。

**Q: 迁移失败了怎么办？**
A: 迁移前自动备份了整个 `.ink/` 目录。最坏情况：恢复备份，和迁移前一模一样。

**Q: 我用的是 Gemini CLI / Codex CLI，也能迁移吗？**
A: 能。迁移工具走的是 Python 脚本，不依赖特定 CLI 平台。

**Q: 迁移后 reader-simulator 每章都跑，会不会太慢？**
A: v9.0 的 reader-simulator 有快速模式（跳过逐段模拟和读者独白），每章只多 1-2 分钟。

**Q: 我可以关掉 Reader Verdict 吗？**
A: 可以。迁移后 `state.json` 里有 `harness_config.reader_verdict_mode`，改成 `"off"` 即可。
